# 02. 사전준비 가이드

설치 전에 먼저 확인할 건 클라우드 이름이 아니라 네트워크 구조야.

---

## 1. 먼저 판단할 것

### NAT 미사용인지 확인

아래에 해당하면 `USE_NAT=false` 쪽으로 보면 된다.

- 서버 NIC에 직접 바인딩되는 고정 IP가 있다
- 그 IP로 K3s API와 서비스 진입점을 직접 잡을 수 있다
- MetalLB로 같은 서브넷 대역의 IP를 할당해도 된다

### NAT 사용인지 확인

아래에 해당하면 `USE_NAT=true` 쪽으로 보면 된다.

- 서버/인스턴스 내부에서는 private IP만 보인다
- 외부에서는 별도 public/external IP가 NAT로 매핑된다
- 서비스 외부 노출은 LoadBalancer보다 NodePort가 현실적이다

## 2. 공통 준비

### 서버 기본 조건

| 항목     | 권장            |
| ------ | ------------- |
| OS     | Rocky Linux 9 |
| CPU    | 최소 2 core     |
| 메모리    | 최소 4 GB       |
| 디스크    | 30 GB 이상      |
| 시간 동기화 | NTP 가능 상태     |

### 필수 파일

```bash
mkdir -p /opt/k3s-lab
cd /opt/k3s-lab
chmod +x k3s-*.sh
```

---

## 3. 포트 준비

### 공통

| 포트 | 프로토콜 | 용도 |
|------|---------|------|
| 6443 | TCP | K3s API Server |
| 80 | TCP | Envoy Gateway HTTP listener (NAT 미사용 환경) |
| 10250 | TCP | kubelet |
| 8472 | UDP | Flannel VXLAN |

### NAT 사용 시 추가 핵심 포트

| 포트 | 프로토콜 | 용도 |
|------|---------|------|
| 30080 | TCP | Envoy Gateway NodePort |

> 사용자 서비스 트래픽은 NAT 사용 환경에서 주로 `30080/tcp`로 들어온다.
> `6443`은 API 서버 포트다.

---

## 4. k3s-common.sh 설정

### NAT 미사용

```bash
USE_NAT=false
DIRECT_BIND_IP="192.168.1.10"
MASTER_HOSTNAME="k3s-master"
```

### NAT 사용, 외부 IP 미확인

```bash
USE_NAT=true
EXTERNAL_IP=""
MASTER_HOSTNAME="k3s-master"
```

이 상태에서는:

- private IP는 자동 감지된다
- K3s는 내부 통신 기준으로 설치 가능하다
- 외부 kubectl 접속과 외부 브라우저 접속은 나중에 설정한다

### NAT 사용, 외부 IP 확인

```bash
USE_NAT=true
EXTERNAL_IP="150.xxx.xxx.xxx"
MASTER_HOSTNAME="k3s-master"
```

## 5. NAT 미사용 환경 추가 준비

`k3s-apps.sh`의 MetalLB IP 풀을 반드시 수정해야 한다.

```bash
METALLB_IP_START="192.168.1.200"
METALLB_IP_END="192.168.1.220"
```

조건:

- 서버 IP와 겹치지 않아야 함
- DHCP 범위와 겹치지 않아야 함
- 실제 같은 L2/LAN에서 사용 가능한 대역이어야 함

---

## 6. NAT 사용 환경 추가 준비

### 외부 IP를 아직 몰라도 되는 범위

다음은 바로 진행 가능하다.

- `k3s-common.sh`
- `k3s-master.sh`
- `k3s-apps.sh`
- 내부망에서 private IP를 hosts에 매핑한 뒤 `도메인:30080` 기준 확인

### 외부 IP를 알아야 가능한 범위

- 외부 PC에서 `kubectl` 원격 접속
- 외부 브라우저에서 `도메인:30080` 접속
- hosts 파일을 external IP 기준으로 배포

### NAT형 환경 체크리스트

- private IP가 노드 내부에서 정상 확인됨
- 보안 그룹/보안 목록에서 `6443/tcp`, `30080/tcp`, `10250/tcp`, `8472/udp` 허용
- 서버 내부 iptables REJECT 룰이 NodePort 통신을 막지 않음

---

## 7. 파일 전송 예시

### NAT 미사용

```bash
scp k3s-*.sh root@192.168.1.10:/opt/k3s-lab/
```

### NAT 사용, 외부 IP를 아는 경우

```bash
scp k3s-*.sh root@<EXTERNAL_IP>:/opt/k3s-lab/
```

### NAT 사용, 외부 IP를 모르는 경우

외부 IP 대신 내부 접속 가능한 경로를 사용하면 된다.

예:

- 사설망 VPN
- 점프 호스트
- 클라우드 콘솔 직렬 접속

---

## 8. 설치 전 최종 체크리스트

### NAT 미사용

- `USE_NAT=false`
- `DIRECT_BIND_IP` 입력 완료
- MetalLB IP 대역 수정 완료
- `6443/tcp`, `10250/tcp`, `8472/udp` 경로 확인

### NAT 사용, 외부 IP 미확인

- `USE_NAT=true`
- `EXTERNAL_IP=""`
- private IP 자동 감지 가능한 상태
- `6443/tcp`, `30080/tcp`, `10250/tcp`, `8472/udp` 정책 확인
- 외부 접속은 나중 반영으로 이해하고 진행

### NAT 사용, 외부 IP 확인

- `USE_NAT=true`
- `EXTERNAL_IP` 입력 완료
- `30080/tcp` 외부 진입 허용
- 외부 kubectl/브라우저 접속까지 바로 검증 가능

## 9. 빠른 트러블슈팅

### NAT 사용 환경에서 NodePort 접속 불가

```bash
ss -tlnp | grep 30080
sudo iptables -L INPUT -n | grep REJECT
```

### private IP 확인

```bash
ip route get 8.8.8.8
```

### API 접속 확인

```bash
curl -k https://<접속 가능한 IP>:6443
```
