## 0. K3s 관리

### K3s 내장 컴포넌트

K3s가 자동으로 설치/관리하는 컴포넌트로, 별도 관리가 필요 없습니다.

| 컴포넌트 | 역할 |
|---------|------|
| containerd | 컨테이너 런타임 |
| Flannel | Pod 네트워크 (CNI) |
| CoreDNS | 클러스터 내부 DNS |
| Metrics Server | `kubectl top` 명령어 지원 |

### K3s 서비스 관리

```bash
systemctl status k3s              # 상태 확인
systemctl restart k3s             # 재시작
journalctl -u k3s -f --no-pager   # 로그 실시간 확인
```

### K3s 업그레이드

**일반 서버:**
```bash
source /etc/k3s-env
curl -sfL https://get.k3s.io | sh -s - server \
  --node-ip "$MASTER_IP" \
  --tls-san "$MASTER_IP" \
  --disable traefik \
  --disable servicelb \
  --write-kubeconfig-mode 644
```

**OCI:**
```bash
source /etc/k3s-env
curl -sfL https://get.k3s.io | sh -s - server \
  --node-ip            "$PRIVATE_IP" \
  --advertise-address  "$PRIVATE_IP" \
  --node-external-ip   "$PUBLIC_IP"  \
  --tls-san            "$PUBLIC_IP"  \
  --tls-san            "$PRIVATE_IP" \
  --disable traefik \
  --disable servicelb \
  --write-kubeconfig-mode 644
```

### K3s 완전 삭제

```bash
/usr/local/bin/k3s-uninstall.sh
```

## 1. MetalLB — 온프레미스 LoadBalancer

> **일반 서버 전용입니다. OCI에서는 설치되지 않습니다.**

### 역할

온프레미스 환경에서 `type: LoadBalancer` 서비스에 External IP를 자동 할당합니다.
Envoy Gateway가 External IP를 받으려면 MetalLB가 필요합니다.

### 상태 확인

```bash
kubectl get ipaddresspool -n metallb-system
kubectl get l2advertisement -n metallb-system
kubectl get svc -A | grep LoadBalancer
kubectl get pods -n metallb-system
```

### IP 대역 변경

```bash
kubectl edit ipaddresspool default-pool -n metallb-system
```

### 트러블슈팅

```bash
# External IP가 <pending>일 때
kubectl logs -n metallb-system -l app.kubernetes.io/component=controller --tail=20
kubectl get ipaddresspool -n metallb-system
```

## 2. Envoy Gateway — 외부 트래픽 진입점

### 환경별 동작 방식

| 환경 | 서비스 타입 | 접속 방법 |
|------|-----------|---------|
| 일반 서버 | LoadBalancer (MetalLB External IP) | `http://도메인` |
| OCI | NodePort (30080) | `http://도메인:30080` |

### 상태 확인

```bash
# 일반 서버: Gateway External IP
kubectl get gateway eg -o jsonpath="{.status.addresses[0].value}"

# OCI: NodePort 확인
kubectl get svc -n envoy-gateway-system
# envoy-* 서비스의 TYPE이 NodePort, PORT에 30080이 있어야 함

# 공통: HTTPRoute 목록
kubectl get httproute
```

### HTTPRoute 관리

```bash
kubectl apply -f my-httproute.yaml    # 추가
kubectl edit httproute <이름>          # 수정
kubectl delete httproute <이름>        # 삭제
```

### [OCI] NodePort 수동 패치 (자동 실패 시)

k3s-apps.sh의 NodePort 패치가 실패한 경우 수동으로 실행합니다.

```bash
# Envoy 서비스 이름 확인
kubectl get svc -n envoy-gateway-system

# NodePort 패치
kubectl patch svc <envoy-서비스명> -n envoy-gateway-system --type merge \
  -p '{"spec":{"type":"NodePort","ports":[{"name":"http","port":80,"targetPort":80,"nodePort":30080,"protocol":"TCP"}]}}'

# 확인
kubectl get svc -n envoy-gateway-system
```

### Helm 설정 변경

```bash
helm get values eg -n envoy-gateway-system
helm upgrade eg oci://docker.io/envoyproxy/gateway-helm \
  --version v1.7.1 -n envoy-gateway-system \
  --set <key>=<value>
```

## 3. cert-manager — TLS 인증서 자동 관리

### 상태 확인

```bash
kubectl get clusterissuer
kubectl get certificate -A
```

### Gateway에 TLS 적용 (Self-Signed)

```yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: eg-default-tls
  namespace: default
spec:
  secretName: eg-default-tls
  issuerRef:
    name: selfsigned-issuer
    kind: ClusterIssuer
  dnsNames:
  - "*.local"
```

### Let's Encrypt로 전환 (도메인 있을 때)

> 공인 도메인이 있을 때만 사용 가능합니다.

```yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: your-email@example.com
    privateKeySecretRef:
      name: letsencrypt-prod
    solvers:
    - http01:
        gatewayHTTPRoute:
          parentRefs:
          - name: eg
            namespace: default
```

> **OCI에서 Let's Encrypt 사용 시**: 80/tcp가 Security List에서 열려 있어야 ACME HTTP-01 챌린지가 작동합니다.

## 4. Headlamp — 클러스터 웹 UI

### 접속

- 일반 서버: `http://dashboard.local`
- OCI: `http://dashboard.local:30080`

### 로그인 토큰

```bash
kubectl -n headlamp create token headlamp
# 출력된 토큰을 브라우저 로그인 화면에 붙여넣기
```

### 수동 재설치 시 패치 필요

```bash
helm upgrade --install headlamp headlamp/headlamp \
  --namespace headlamp --set image.tag="v0.40.1"

kubectl -n headlamp patch deployment headlamp --type='json' -p='[
  {"op": "replace", "path": "/spec/template/spec/containers/0/args", "value": [
    "-in-cluster",
    "-in-cluster-context-name=main",
    "-plugins-dir=/headlamp/plugins"
  ]}
]'
```

## 5. Prometheus + Grafana — 메트릭 모니터링

### Grafana 접속

- 일반 서버: `http://grafana.local` (admin / admin1234)
- OCI: `http://grafana.local:30080` (admin / admin1234)

### 커스텀 ServiceMonitor 추가

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: my-app-monitor
  namespace: monitoring
  labels:
    release: kube-prometheus-stack
spec:
  namespaceSelector:
    matchNames: [default]
  selector:
    matchLabels:
      app: my-app
  endpoints:
  - port: metrics
    interval: 30s
```

### 설정 변경

```bash
# Grafana 비밀번호 변경
helm upgrade kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  -n monitoring --set grafana.adminPassword=새비밀번호

# Prometheus retention 변경
helm upgrade kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  -n monitoring --set prometheus.prometheusSpec.retention=14d
```

## 6. Loki + Alloy — 로그 수집/조회

### 현재 구성

- **Loki**: SingleBinary + filesystem (emptyDir)
- **Alloy**: DaemonSet으로 모든 노드의 Pod 로그 수집

> emptyDir이므로 Loki Pod 재시작 시 로그가 초기화됩니다. 로그 영속성이 필요하면 PV 연결이 필요합니다.

### Grafana Loki 데이터소스

k3s-apps.sh 설치 시 ConfigMap 자동 프로비저닝으로 **수동 추가 없이 자동 등록**됩니다.

```bash
# 등록 확인
kubectl get configmap loki-datasource -n monitoring
```

> 데이터소스가 보이지 않으면 Grafana Pod를 재시작하세요:
> `kubectl rollout restart deployment kube-prometheus-stack-grafana -n monitoring`

Grafana에서 확인: Connections → Data Sources → Loki
URL: `http://loki-gateway.logging.svc.cluster.local`

### LogQL 예시

```
{namespace="default"} |= "error"
{namespace="default"} | json | level="ERROR"
rate({namespace="default"}[5m])
{pod="my-app-xxxxx"} | line_format "{{.msg}}"
```

### 상태 확인

```bash
kubectl get pods -n logging -l app.kubernetes.io/name=loki
kubectl get pods -n logging -l app.kubernetes.io/name=alloy
kubectl logs -n logging -l app.kubernetes.io/name=loki --tail=20
```

## 7. Sealed Secrets — GitOps Secret 암호화

### 사용법

```bash
# 1. Secret 생성 (dry-run으로 YAML만 추출)
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password=super-secret-pw \
  --dry-run=client -o yaml > db-secret.yaml

# 2. 암호화 (Sealed Secret 생성)
kubeseal --format yaml < db-secret.yaml > db-sealed-secret.yaml

# 3. Git에 커밋 & 적용
kubectl apply -f db-sealed-secret.yaml
```

### 공개키 백업 (중요!)

```bash
# 클러스터 분실/재설치 시 SealedSecret 복호화 불가
# 반드시 공개키를 외부에 백업
kubeseal --fetch-cert > sealed-secrets-cert.pem
```

## 8. Argo CD — GitOps 배포

### 접속

- 일반 서버: `http://argocd.local`
- OCI: `http://argocd.local:30080`

```bash
# 초기 비밀번호
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d; echo
```

### UI에서 Application 생성

1. **Applications** > **+ NEW APP**
2. **GENERAL**: Application Name, Sync Policy (Manual / Automatic)
3. **SOURCE**: Repository URL, Revision (브랜치/태그), Path (Helm/Kustomize 경로)
4. **DESTINATION**: `https://kubernetes.default.svc`, Namespace
5. **CREATE** → Sync 클릭

### 배포 흐름

```
Git push
  → ArgoCD 감지 (기본 3분 폴링, 또는 Git Webhook)
  → Diff 표시
  → 자동(Auto-Sync) 또는 수동(Sync 버튼) 배포
  → 클러스터 반영
```

### 설정 변경

```bash
# Sync 주기 변경 (기본 3분 → 1분)
kubectl edit configmap argocd-cm -n argocd
# data에 추가: timeout.reconciliation: 60s

# TLS 활성화 (현재 insecure 모드)
kubectl patch configmap argocd-cmd-params-cm -n argocd --type merge \
  -p '{"data":{"server.insecure":"false"}}'
kubectl rollout restart deployment argocd-server -n argocd
```

## 9. Argo CD Image Updater — 이미지 자동 업데이트

### 역할

컨테이너 레지스트리를 모니터링하다가 새 이미지가 올라오면
자동으로 Git 커밋 또는 ArgoCD Application을 업데이트합니다.

### Annotation 설정 (ArgoCD Application에 추가)

```yaml
metadata:
  annotations:
    argocd-image-updater.argoproj.io/image-list: myapp=your-registry/my-app
    argocd-image-updater.argoproj.io/myapp.update-strategy: semver
    argocd-image-updater.argoproj.io/myapp.allow-tags: regexp:^1\.[0-9]+\.[0-9]+$
```

### 로그 확인

```bash
kubectl -n argocd logs -l app.kubernetes.io/name=argocd-image-updater -f
```
