# 03. 설치 단계별 가이드

## 전체 흐름

```text
[1] k3s-common.sh  → OS 공통 설정
[2] k3s-master.sh  → K3s 서버 설치
[3] k3s-apps.sh    → DevOps 스택 설치
[4] 접속 확인      → API / 브라우저 / HTTPRoute 확인
```

---

## 0단계: 설정값 정리

### NAT 미사용

`k3s-common.sh`

```bash
USE_NAT=false
DIRECT_BIND_IP="192.168.1.10"
```

`k3s-apps.sh`

```bash
METALLB_IP_START="192.168.1.200"
METALLB_IP_END="192.168.1.220"
```

### NAT 사용, 외부 IP 미확인

`k3s-common.sh`

```bash
USE_NAT=true
EXTERNAL_IP=""
```

### NAT 사용, 외부 IP 확인

`k3s-common.sh`

```bash
USE_NAT=true
EXTERNAL_IP="150.xxx.xxx.xxx"
```

## 1단계: OS 공통 설정

```bash
sudo bash k3s-common.sh
```

### 결과

- NAT 미사용: `DIRECT_BIND_IP` 기준으로 `/etc/k3s-env` 저장
- NAT 사용: `PRIVATE_IP` 자동 감지 후 `/etc/k3s-env` 저장
- 외부 IP를 모르는 NAT 환경도 중단하지 않고 계속 진행

예상 저장값:

### NAT 미사용

```bash
USE_NAT=false
DIRECT_BIND_IP=192.168.1.10
```

### NAT 사용

```bash
USE_NAT=true
PRIVATE_IP=10.0.0.10
EXTERNAL_IP=
```

## 2단계: K3s 서버 설치

```bash
sudo bash k3s-master.sh
```

### master.sh가 먼저 확인하는 것

`k3s-master.sh`는 시작할 때 `/etc/k3s-env`를 먼저 확인한다.

- 파일이 있고 필수 값이 정상이라면 그대로 진행
- 파일이 없거나 비정상이라면 같은 디렉터리의 `k3s-common.sh`를 자동 실행
- `k3s-common.sh` 실행 후 `/etc/k3s-env`를 다시 검증하고 나서 설치 진행

즉 `k3s-cleanup.sh`로 삭제한 직후에도 아래처럼 바로 다시 시작할 수 있다.

```bash
sudo bash k3s-master.sh
```

단, 이 경우에도 `k3s-common.sh` 상단 설정값은 먼저 맞춰놔야 한다.

### 수동 확인 절차

재설치 전에 직접 확인하고 싶다면:

```bash
ls -l /etc/k3s-env
cat /etc/k3s-env
```

파일이 없으면 정상이다.
`k3s-cleanup.sh`가 지운 상태이기 때문에 `k3s-master.sh`가 `k3s-common.sh`를 자동 실행해서 다시 생성한다.

### 설치 옵션 요약

| 상태 | 주요 옵션 |
|------|-----------|
| NAT 미사용 | `--node-ip DIRECT_BIND_IP`, `--tls-san DIRECT_BIND_IP` |
| NAT 사용, 외부 IP 미확인 | `--node-ip PRIVATE_IP`, `--advertise-address PRIVATE_IP`, `--tls-san PRIVATE_IP` |
| NAT 사용, 외부 IP 확인 | 위 옵션 + `--node-external-ip EXTERNAL_IP`, `--tls-san EXTERNAL_IP` |

### 확인

```bash
kubectl get nodes -o wide
```

해석:

- NAT 미사용: `INTERNAL-IP`가 직접 바인딩 IP로 보이면 정상
- NAT 사용, 외부 IP 미확인: `INTERNAL-IP`만 보여도 정상
- NAT 사용, 외부 IP 확인: `EXTERNAL-IP`에 외부 IP가 보이면 정상

## NAT 사용 + 외부 IP 확인 후 추가 반영

외부 IP를 나중에 확인했다면 아래를 순서대로 반영한다.

1. `/etc/k3s-env`의 `EXTERNAL_IP` 반영
2. `k3s-master.sh`를 다시 실행하거나 동일 옵션으로 K3s server 재반영
3. 워커가 있다면 `k3s-worker.sh`도 external IP 기준으로 재반영
4. kubeconfig의 `server:` 주소를 `https://EXTERNAL_IP:6443`로 치환
5. hosts 파일을 external IP 기준으로 갱신

상세 절차는 [07-외부-IP-나중-반영-가이드.md](/Users/taekyu/Documents/git/server-Create/app-server/install/k3s/07-외부-IP-나중-반영-가이드.md) 참고.

## 원격 kubeconfig 설정

### NAT 사용 + 외부 IP 확인 후

```bash
scp root@<EXTERNAL_IP>:/etc/rancher/k3s/k3s.yaml ~/.kube/config
sed -i "s/127.0.0.1/<EXTERNAL_IP>/g" ~/.kube/config
kubectl get nodes
```

> 외부 IP가 없다면 이 단계는 아직 보류해야 한다.

## 전체 삭제

### 단일 노드 정리

```bash
sudo bash k3s-cleanup.sh
```

확인 없이 바로 실행하고, `helm`과 `kubeseal`까지 지우려면:

```bash
sudo bash k3s-cleanup.sh --yes --remove-tools
```

옵션 차이:

- `sudo bash k3s-cleanup.sh`
  현재 노드의 K3s와 로컬 설정을 지우고, 실행 전에 확인 질문을 한 번 묻는다.
- `sudo bash k3s-cleanup.sh --yes`
  위와 같지만 확인 질문 없이 바로 진행한다.
- `sudo bash k3s-cleanup.sh --remove-tools`
  기본 동작에 더해 `helm`, `kubeseal`도 같이 지운다. 확인 질문은 그대로 묻는다.
- `sudo bash k3s-cleanup.sh --yes --remove-tools`
  확인 질문 없이 바로 진행하고 `helm`, `kubeseal`도 같이 지운다.

이 스크립트는 아래를 정리한다.

- 현재 노드의 `k3s` 또는 `k3s-agent`
- `/etc/k3s-env`
- k3s용 sysctl / modules 설정
- `/etc/hosts`의 `k3s-master` 항목
- root와 설치 사용자 `.bashrc`에 추가한 KUBECONFIG / kubectl alias
- K3s 관련 잔여 디렉터리

자동으로 되돌리지 않는 항목:

- root 비밀번호
- SELinux 설정
- firewalld 상태
- timezone / locale

재설치할 때는:

```bash
sudo bash k3s-master.sh
```

이 명령 하나로 `/etc/k3s-env` 존재 여부를 확인하고, 없으면 `k3s-common.sh`를 먼저 실행한 뒤 설치를 이어간다.

### 워커가 있는 클러스터 삭제 순서

1. 마스터에서 워커를 먼저 drain / delete
2. 각 워커에서 `sudo bash k3s-cleanup.sh`
3. 마지막에 마스터에서 `sudo bash k3s-cleanup.sh`

예:

```bash
kubectl drain k3s-worker1 --ignore-daemonsets --delete-emptydir-data
kubectl delete node k3s-worker1

kubectl drain k3s-worker2 --ignore-daemonsets --delete-emptydir-data
kubectl delete node k3s-worker2
```
