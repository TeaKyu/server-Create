# 01. 프로젝트 개요 및 아키텍처

이 스택은 단일 K3s 마스터 기준으로 DevOps 도구를 한 번에 설치하는 구성이야.
핵심 기준은 특정 클라우드 이름이 아니라 네트워크 구조다.

이 저장소는 아래 두 가지를 기준으로 동작한다.

1. `NAT 미사용`
2. `NAT 사용`

NAT 사용 환경에서는 외부 IP를 이미 알고 있는지 여부도 함께 본다.

---

## 환경 분류

| 항목 | NAT 미사용 | NAT 사용 |
|------|------------|----------|
| 노드가 직접 바인딩하는 IP | `DIRECT_BIND_IP` | `PRIVATE_IP` |
| 외부에서 보이는 주소 | 보통 같은 IP 또는 별도 LB | `EXTERNAL_IP` |
| K3s API 바인딩 기준 | 직접 바인딩 IP | private IP |
| LoadBalancer 구현 | MetalLB | ServiceLB (k3s 내장) |
| 사용자 서비스 진입점 | MetalLB External IP | ServiceLB (표준 포트 80/443) |
| 외부 IP 미확인 상태 | 해당 없음 | 내부 설치만 먼저 가능 |

> NAT 사용 환경은 VM/인스턴스 NIC에는 private IP만 있고, 외부에서는 NAT를 통해 별도 external IP로 접속하는 구조를 말한다.

---

## 트래픽 구조

### NAT 미사용

```text
브라우저 / kubectl
        │
        ├── API: https://DIRECT_BIND_IP:6443
        └── 서비스: http://도메인
                    │
                    └── MetalLB External IP
                        → Envoy Gateway
                        → HTTPRoute
                        → 각 서비스
```

### NAT 사용

```text
브라우저 / kubectl
        │
        ├── API: 내부는 https://PRIVATE_IP:6443
        │       외부는 https://EXTERNAL_IP:6443 (알고 있을 때만)
        │
        └── 서비스: http(s)://도메인
                    │
                    └── 공유기 포트포워딩 (외부 80/443 → 서버 80/443)
                        → ServiceLB (노드 Private IP:80/443)
                        → Envoy Gateway
                        → HTTPRoute
                        → 각 서비스
```

> `6443`은 K3s API 서버 포트고, 사용자 서비스 포트가 아니다.
> 사용자 서비스는 Envoy Gateway를 통해 처리된다.

## 주요 스크립트

| 파일 | 역할 | 주요 설정값 |
|------|------|-------------|
| `k3s-common.sh` | OS 공통 설정과 `/etc/k3s-env` 생성 | `USE_NAT`, `DIRECT_BIND_IP`, `EXTERNAL_IP` |
| `k3s-master.sh` | K3s 서버 설치 | `/etc/k3s-env` 로드 |
| `k3s-apps.sh` | DevOps 스택 설치 | `METALLB_IP_START`, `METALLB_IP_END` |
| `k3s-worker.sh` | 워커 조인 | `K3S_TOKEN`, `MASTER_DIRECT_IP`, `MASTER_PRIVATE_IP` |
| `k3s-cleanup.sh` | 현재 노드의 K3s와 로컬 설정 정리 | `--yes`, `--remove-tools` |

## 외부 IP 상태별 해석

### NAT 사용 + 외부 IP 미확인

- `k3s-common.sh`는 계속 진행한다
- `k3s-master.sh`는 private IP 기준으로 설치하고 ServiceLB를 활성화한다
- `k3s-apps.sh`는 ServiceLB로 표준 포트(80/443) 기준으로 서비스를 준비한다
- 공유기 포트포워딩(외부 80/443 → 서버 80/443) 설정이 별도로 필요하다
- 외부 인터넷 접속과 kubeconfig 외부 주소 치환은 나중에 반영한다

### NAT 사용 + 외부 IP 확인

- K3s 서버에 `--node-external-ip`와 `--tls-san EXTERNAL_IP`를 추가한다
- kubeconfig를 `https://EXTERNAL_IP:6443` 기준으로 외부에서 사용할 수 있다
- 사용자 서비스는 `http(s)://도메인` (표준 포트) 으로 외부 접속할 수 있다

---

## 운영 관점 요약

- 내부 제어면 통신은 NAT 사용 여부와 무관하게 실제 노드 IP 기준으로 본다
- NAT 사용 환경은 ServiceLB가 노드 IP를 External IP로 할당하고, 공유기에서 80/443 포트포워딩으로 연결한다
- 워커 조인은 NAT 사용 시에도 마스터의 private IP 기준으로 진행한다
- 외부 IP를 나중에 알아도 사용자 서비스 구조 자체는 그대로 유지된다
