# 06. 워커 노드 확장 가이드

마스터 1대로 운영 중인 K3s 클러스터에 워커 노드를 추가하는 방법입니다.
워커를 추가하면 Pod가 여러 노드에 분산 스케줄링되어 리소스를 효율적으로 사용할 수 있습니다.

---

## 워커 추가 시 달라지는 점

| 항목 | 마스터 1대 (현재) | 마스터 + 워커 2대 |
|------|-----------------|------------------|
| Pod 스케줄링 | 마스터에 모든 Pod 실행 | 워커에 분산 실행 |
| 리소스 | 마스터 1대에 집중 | 여러 노드로 분산 |
| 가용성 | 마스터 다운 시 전체 중단 | 워커 1대 다운해도 서비스 유지 |
| Alloy (로그 수집) | 마스터에서만 수집 | 각 워커 노드에서도 수집 |

---

## 확장 구성

| 노드 | 호스트명 | Private IP | Public IP | 최소 사양 | 역할 |
|------|---------|-----------|-----------|----------|------|
| Master | k3s-master | (자동 감지) | (OCI 콘솔) | 2 core / 4 GB | Server (Control Plane) |
| Worker 1 | k3s-worker1 | (자동 감지) | (OCI 콘솔) | 2 core / 4 GB | Agent (워크로드 실행) |
| Worker 2 | k3s-worker2 | (자동 감지) | (OCI 콘솔) | 2 core / 4 GB | Agent (워크로드 실행) |

> 같은 VCN 서브넷 내의 인스턴스끼리는 Private IP로 통신합니다.
> 각 워커의 Public IP는 OCI 콘솔에서 확인하여 k3s-common.sh에 입력합니다.

### 네트워크 구성 (OCI 환경)

```
클라이언트 PC (브라우저)
    │
    └── 인터넷
            │
            ├── OCI Public IP (마스터) → NAT → k3s-master (Private IP)
            │                                   ↑↓ VCN 내부 통신
            ├── OCI Public IP (워커1) → NAT → k3s-worker1 (Private IP)
            ├── OCI Public IP (워커2) → NAT → k3s-worker2 (Private IP)
            │
            └── Envoy Gateway (NodePort 30080, 마스터에서 서비스)
```

### 추가 포트 요구사항

마스터-워커 간 통신에 필요한 포트입니다.

| 포트 | 프로토콜 | 용도 | 방향 |
|------|---------|------|------|
| 6443 | TCP | K3s API Server | 워커 → 마스터 |
| 10250 | TCP | kubelet | 마스터 ↔ 워커 |
| 8472 | UDP | Flannel VXLAN | 노드 간 양방향 |

## 워커 추가 절차

### 1단계 — 워커 서버에 OS 설정

각 워커 서버에 `k3s-common.sh`를 복사하고, **환경변수에 워커 hosts를 추가**한 후 실행합니다.

워커용으로 `k3s-common.sh`를 수정하세요:

```bash
# k3s-common.sh 상단의 환경 변수를 워커의 Public IP로 수정
PUBLIC_IP="<워커의 OCI Public IP>"    # OCI 콘솔에서 확인
MASTER_HOSTNAME="k3s-master"
```

> Private IP는 자동 감지되므로 수정할 필요 없습니다.

각 워커에서 실행:

```bash
# Worker 1 (워커1의 OCI Public IP로 SSH 접속)
ssh root@<워커1_PUBLIC_IP>
scp root@<마스터_PRIVATE_IP>:/opt/k3s-lab/k3s-common.sh /opt/k3s-lab/
# PUBLIC_IP를 워커1의 Public IP로 수정 후 실행
vi /opt/k3s-lab/k3s-common.sh
sudo bash /opt/k3s-lab/k3s-common.sh
```

### 2단계 — 마스터에서 조인 토큰 확인

```bash
# 마스터 서버에서 실행
cat /var/lib/rancher/k3s/server/node-token
```

출력 예시:

```
K10abc123def456ghi789jkl012mno345pqr678stu901::server:vwx234yz567abc890def
```

이 전체 문자열을 복사합니다.

### 3단계 — 워커 스크립트 설정

`k3s-worker.sh`를 각 워커에 복사하고 환경변수를 수정합니다:

```bash
# k3s-worker.sh 상단 수정
MASTER_PRIVATE_IP="<마스터의 Private IP>"   # 마스터의 Private IP (같은 VCN 통신)
K3S_TOKEN="K10abc123..."                      # 위에서 복사한 토큰 전체
```

> OCI 환경에서는 마스터의 **Private IP**를 사용합니다 (같은 VCN 내 통신).
> 워커 자신의 Private/Public IP는 `/etc/k3s-env`에서 자동으로 읽어옵니다.

### 4단계 — 워커 조인

```bash
# Worker 1
ssh root@<워커1_PUBLIC_IP>
sudo bash /opt/k3s-lab/k3s-worker.sh

# Worker 2
ssh root@<워커2_PUBLIC_IP>
sudo bash /opt/k3s-lab/k3s-worker.sh
```

### 5단계 — 조인 확인

마스터에서 실행:

```bash
kubectl get nodes -o wide
# NAME          STATUS   ROLES                  AGE   VERSION
# k3s-master    Ready    control-plane,master   1h    v1.32.x+k3s1
# k3s-worker1   Ready    <none>                 2m    v1.32.x+k3s1
# k3s-worker2   Ready    <none>                 1m    v1.32.x+k3s1
```

3대 모두 `Ready` 상태인지 확인합니다.

## 워커 추가 후 확인사항

### Pod 분산 확인

워커를 추가하면 새로 생성되는 Pod는 자동으로 분산됩니다.
기존 Pod는 직접 재시작해야 재분배됩니다:

```bash
# 특정 Deployment 재시작 (Pod 재분배)
kubectl rollout restart deployment <이름> -n <네임스페이스>

# 전체 Pod의 노드 배치 확인
kubectl get pods -A -o wide | grep -v Completed
```

### Alloy (로그 수집) DaemonSet 확인

Alloy는 DaemonSet이므로 워커가 추가되면 자동으로 각 노드에 Pod가 생성됩니다:

```bash
kubectl get pods -n logging -l app.kubernetes.io/name=alloy -o wide
# 노드 수만큼 Pod가 있어야 함 (마스터 1 + 워커 2 = 3개)
```

## 워커 제거

워커 노드를 클러스터에서 제거하려면:

### 1단계 — 마스터에서 노드 drain + 삭제

```bash
# Pod를 다른 노드로 이동 (graceful)
kubectl drain k3s-worker1 --ignore-daemonsets --delete-emptydir-data

# 클러스터에서 노드 제거
kubectl delete node k3s-worker1
```

### 2단계 — 워커에서 K3s 삭제

```bash
ssh root@192.168.1.11
/usr/local/bin/k3s-agent-uninstall.sh
```

## 워커 조인 실패 시 트러블슈팅

### 토큰 오류

```bash
# 마스터에서 새 토큰 확인
cat /var/lib/rancher/k3s/server/node-token

# 워커에서 K3s Agent 삭제 후 재시도
/usr/local/bin/k3s-agent-uninstall.sh
# k3s-worker.sh의 K3S_TOKEN 수정 후 재실행
sudo bash k3s-worker.sh
```

### 네트워크 연결 오류

```bash
# 워커에서 마스터 API 접근 확인 (Private IP 사용)
curl -k https://<마스터_PRIVATE_IP>:6443
# 응답이 오면 네트워크 OK

# 방화벽 확인
systemctl status firewalld
# active이면 비활성화 필요

# OCI iptables REJECT 룰 확인
iptables -L INPUT -n | grep REJECT
# REJECT 룰이 있으면 k3s-common.sh를 다시 실행
```

### K3s Agent 로그 확인

```bash
# 워커에서 실행
journalctl -u k3s-agent -f --no-pager
```

### DNS 해석 문제

```bash
# 워커에서 마스터 호스트명 해석 확인
ping k3s-master
# 실패하면 /etc/hosts에 마스터 IP가 등록되어 있는지 확인
cat /etc/hosts | grep k3s-master
```

## 마스터에서 워크로드 실행 방지 (선택)

워커를 추가한 후 마스터에서는 인프라 컴포넌트만 실행하고 싶다면,
마스터에 taint를 추가합니다:

```bash
# 마스터에 taint 추가 (새 Pod는 워커에만 스케줄링)
kubectl taint nodes k3s-master node-role.kubernetes.io/master:NoSchedule

# taint 해제 (다시 마스터에서도 실행 허용)
kubectl taint nodes k3s-master node-role.kubernetes.io/master:NoSchedule-
```

> 기존에 마스터에서 실행 중인 Pod는 영향받지 않습니다.
> 새로 생성되는 Pod만 워커에 스케줄링됩니다.

## k3s-worker.sh 스크립트 참조

워커 조인용 스크립트 `k3s-worker.sh`의 환경변수를 수정한 후 사용하세요.

```
파일: k3s-worker.sh
위치: /opt/k3s-lab/k3s-worker.sh
```

환경변수:

```bash
MASTER_PRIVATE_IP=""   # 마스터의 Private IP (같은 VCN 내 통신)
K3S_TOKEN=""           # 마스터에서 확인한 토큰
```

> 워커 자신의 Private/Public IP는 k3s-common.sh에서 생성한 `/etc/k3s-env`에서 자동으로 읽어옵니다.
