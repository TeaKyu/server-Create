# 07. 외부 IP 나중 반영 가이드

이 문서는 NAT 사용 환경에서 처음에는 `EXTERNAL_IP=""` 상태로 설치했다가,
나중에 외부 IP를 알게 되었을 때 무엇을 어떻게 반영하는지 상세히 설명한다.

---

## 1. 언제 이 작업이 필요한가

처음 설치할 때 아래처럼 진행한 경우가 대상이다.

```bash
USE_NAT=true
EXTERNAL_IP=""
```

이 상태에서는:

- K3s와 앱 스택 설치 가능
- 내부 통신과 내부 확인 가능
- 외부 인터넷에서 API와 브라우저 접속은 아직 미완성

나중에 외부 IP를 알게 되면 아래 두 가지를 완성하는 작업이라고 보면 된다.

1. K3s API 외부 접속
2. 사용자 서비스 외부 접속

---

## 2. 영향 범위

### 반영 후 달라지는 것

- `kubectl get nodes -o wide` 에서 master의 `EXTERNAL-IP`가 보일 수 있다
- 외부 PC에서 `https://EXTERNAL_IP:6443` 로 kubectl 접속 가능
- 외부 브라우저에서 `http(s)://도메인` 접속 가능 (공유기 포트포워딩 필요)
- hosts 파일을 external IP 기준으로 배포 가능

### 반영 안 해도 그대로인 것

- 노드 내부 통신
- Pod 간 통신
- 내부 private IP 기준 접속
- 이미 설치된 앱 스택 자체

즉 이 작업은 "서비스 구조 변경"이 아니라 "외부 진입 주소 완성" 작업이다.

---

## 3. 먼저 확인할 것

마스터에서:

```bash
cat /etc/k3s-env
kubectl get nodes -o wide
kubectl get svc -n envoy-gateway-system
```

확인 포인트:

- `USE_NAT=true` 여야 함
- `PRIVATE_IP` 값이 있어야 함
- `EXTERNAL_IP` 는 비어 있거나 아직 미설정 상태일 수 있음
- Envoy 서비스가 `LoadBalancer` 타입이고 `EXTERNAL-IP`에 서버 IP가 할당되어야 함
- ServiceLB 파드(`svclb-*`)가 kube-system 네임스페이스에 실행 중이어야 함

```bash
kubectl get pods -n kube-system | grep svclb
kubectl get svc -n envoy-gateway-system
```

외부 정책도 같이 확인:

- `6443/tcp` 외부 허용 (kubectl 원격 접속용)
- `80/tcp`, `443/tcp` 외부 허용 + 공유기 포트포워딩 설정

---

## 4. 마스터에 외부 IP 반영

이 단계는 사실상 필수다.

### 4-1. `/etc/k3s-env` 업데이트

마스터에서:

```bash
sudo sed -i 's/^EXTERNAL_IP=.*/EXTERNAL_IP=150.xxx.xxx.xxx/' /etc/k3s-env
grep EXTERNAL_IP /etc/k3s-env
```

### 4-2. 작업 디렉터리의 `k3s-common.sh`도 같이 맞춰두기

이건 즉시 반영에 필수는 아니지만 권장이다.
나중에 `cleanup` 후 재설치할 때 다시 헷갈리지 않게 맞춰두는 용도다.

```bash
vi /opt/k3s-lab/k3s-common.sh
```

설정 구간 예:

```bash
USE_NAT=true
EXTERNAL_IP="150.xxx.xxx.xxx"
```

### 4-3. `k3s-master.sh` 재실행

`k3s-master.sh`는 `/etc/k3s-env`를 읽어서
`EXTERNAL_IP`가 있으면 자동으로 `--node-external-ip` 와 `--tls-san EXTERNAL_IP`를 포함한다.

```bash
sudo bash /opt/k3s-lab/k3s-master.sh
```

### 4-4. 확인

```bash
kubectl get nodes -o wide
```

정상 기대:

- master의 `EXTERNAL-IP` 칼럼에 외부 IP 표시
- 서비스 상태 정상

추가 확인:

```bash
systemctl status k3s --no-pager
journalctl -u k3s -n 50 --no-pager
```

---

## 5. 워커에 외부 IP 반영

이 단계는 선택이다.

### 언제 필요한가

- `kubectl get nodes -o wide` 에서 워커 `EXTERNAL-IP`도 정확히 보이게 하고 싶을 때
- 워커 노드도 외부에서 직접 식별해야 할 때

### 언제 생략 가능하나

- 현재 저장소 기준처럼 사용자 서비스 외부 진입을 master 쪽 IP로만 쓸 때
- 워커 external IP가 없어도 운영상 문제 없을 때

즉 대부분의 단순 운영은 master만 반영해도 충분하다.

### 워커 반영 절차

각 워커에서:

```bash
sudo sed -i 's/^EXTERNAL_IP=.*/EXTERNAL_IP=150.xxx.xxx.xxx/' /etc/k3s-env
grep EXTERNAL_IP /etc/k3s-env
```

그리고 워커의 `k3s-common.sh`도 가능하면 맞춰둔다.

```bash
vi /opt/k3s-lab/k3s-common.sh
```

그 다음 `k3s-worker.sh` 설정 구간에 아래 값이 들어 있어야 한다.

```bash
K3S_TOKEN="<마스터 토큰>"
MASTER_PRIVATE_IP="<마스터 private IP>"
```

재반영:

```bash
sudo bash /opt/k3s-lab/k3s-worker.sh
```

마스터에서 확인:

```bash
kubectl get nodes -o wide
```

## 6. 원격 kubeconfig 갱신

이건 외부 PC에서 kubectl을 직접 쓰고 싶을 때만 필요하다.

중요:

- 서버 내부의 `/etc/rancher/k3s/k3s.yaml` 은 로컬용이라 `127.0.0.1` 이어도 정상
- 외부 PC에서 사용할 복사본만 `EXTERNAL_IP`로 바꾸면 된다

외부 PC에서:

```bash
mkdir -p ~/.kube
scp root@<EXTERNAL_IP>:/etc/rancher/k3s/k3s.yaml ~/.kube/config
sed -i "s/127.0.0.1/<EXTERNAL_IP>/g" ~/.kube/config
kubectl get nodes
```

기대 결과:

- 인증서 오류 없이 접속
- `kubectl get nodes` 성공

만약 인증서 오류가 난다면:

- `k3s-master.sh`가 external IP 포함 상태로 재실행되지 않았을 가능성이 큼
- 즉 `--tls-san EXTERNAL_IP` 가 아직 안 들어간 상태일 수 있음

## 7. 브라우저 접속용 hosts 갱신

외부 PC에서 hosts를 external IP 기준으로 바꾼다.

```text
150.xxx.xxx.xxx  grafana.local
150.xxx.xxx.xxx  prometheus.local
150.xxx.xxx.xxx  argocd.local
150.xxx.xxx.xxx  dashboard.local
```

접속 주소 (공유기 포트포워딩 완료 후):

- `http://grafana.local`
- `http://prometheus.local`
- `http://argocd.local`
- `http://dashboard.local`

중요:

- ServiceLB가 표준 포트(80/443)를 바인딩하므로 포트 번호 없이 접속한다
- Host 기반 라우팅이라서 IP만 직접 치는 방식보다 도메인 기준 접속이 맞다
- 공유기에서 외부 80 → 서버 80, 외부 443 → 서버 443 포트포워딩이 설정되어야 한다

---

## 8. 전체 검증 순서

### 1. 마스터 재반영 확인

```bash
grep EXTERNAL_IP /etc/k3s-env
kubectl get nodes -o wide
systemctl status k3s --no-pager
```

### 2. Gateway / ServiceLB 확인

```bash
# ServiceLB 파드 상태
kubectl get pods -n kube-system | grep svclb

# Envoy 서비스 External IP 확인
kubectl get svc -n envoy-gateway-system

# 표준 포트 바인딩 확인
ss -tlnp | grep -E '":80|":443'
```

### 3. HTTPRoute 확인

```bash
kubectl get httproute
kubectl get gateway eg
```

### 4. 외부 API 확인

외부 PC에서:

```bash
kubectl get nodes
```

### 5. 외부 브라우저 확인

외부 PC 브라우저에서:

- `http://grafana.local`
- `http://argocd.local`

## 9. 가장 흔한 실패 원인

### 1. `EXTERNAL-IP`가 계속 비어 있음

원인:

- `/etc/k3s-env`만 바꾸고 `k3s-master.sh`를 다시 안 돌림
- 워커는 별도 반영 안 함

조치:

```bash
sudo bash /opt/k3s-lab/k3s-master.sh
```

워커도 필요하면:

```bash
sudo bash /opt/k3s-lab/k3s-worker.sh
```

### 2. 외부 kubectl에서 인증서 오류

원인:

- `--tls-san EXTERNAL_IP` 가 반영되지 않음

조치:

- `/etc/k3s-env`의 `EXTERNAL_IP` 확인
- `k3s-master.sh` 재실행
- kubeconfig를 다시 복사해서 교체

### 3. 브라우저 접속 안 됨

원인 후보:

- `80/tcp`, `443/tcp` 외부 오픈 안 됨
- 공유기 포트포워딩 미설정 (외부 80/443 → 서버 80/443)
- hosts 파일이 old IP 그대로임
- ServiceLB 파드가 실행 중이 아님

조치:

```bash
kubectl get pods -n kube-system | grep svclb
kubectl get svc -n envoy-gateway-system
ss -tlnp | grep -E '":80|":443'
```

### 4. `http://EXTERNAL_IP` 로는 되는데 도메인으로는 안 됨

원인:

- hosts 파일이 안 맞음
- Host 기반 라우팅이어서 도메인 매핑이 필요함

조치:

- hosts 파일 갱신
- `grafana.local`, `argocd.local` 같은 도메인으로 접속

---

## 10. 권장 순서 요약

가장 안전한 순서는 이거다.

1. 마스터 `/etc/k3s-env` 의 `EXTERNAL_IP` 업데이트
2. 마스터 `k3s-common.sh` 설정도 같이 맞춤
3. `sudo bash /opt/k3s-lab/k3s-master.sh`
4. 외부 PC kubeconfig 갱신
5. 외부 PC hosts 갱신
6. 브라우저와 kubectl 검증
7. 필요할 때만 워커 external IP 반영

이 저장소 기준으로는 보통 master만 먼저 반영해도 외부 서비스 접속까지는 충분하다.
