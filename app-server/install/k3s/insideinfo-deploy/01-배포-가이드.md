# insideinfo 배포 가이드

k3s + k3s-apps.sh 실행 완료 상태에서 insideinfo 앱(API + Web)을 배포하는 단계별 가이드.

---

## 전제 조건

| 항목             | 상태                                     |
| -------------- | -------------------------------------- |
| k3s 클러스터       | k3s-master.sh + k3s-apps.sh 실행 완료      |
| Argo CD        | 설치됨 (argocd 네임스페이스)                    |
| Envoy Gateway  | 설치됨 (envoy-gateway-system)             |
| Sealed Secrets | 설치됨 (kube-system)                      |
| Jenkins        | Docker 이미지 빌드 → GitLab registry push 중 |
| 외부 IP          | 아직 없음 (내부 테스트만)                        |

## Step 3: GitLab 레지스트리 imagePullSecret 생성

Jenkins가 push한 이미지를 k8s가 pull하려면 GitLab 레지스트리 인증 정보가 필요하다.

```bash
kubectl create secret docker-registry gitlab-registry-secret \
  --docker-server=<GITLAB_REGISTRY> \        # gitlab.insideinfo.co.kr:5005
  --docker-username=<GITLAB_USER> \          # qwep0224
  --docker-password=<GITLAB_TOKEN> \         # GitLab Personal Access Token
  -n insideinfo
```

> **팁**: GitLab Personal Access Token (scope: `read_registry, read_repository`)을 사용하는 것을 권장.

확인:

```bash
kubectl get secret gitlab-registry-secret -n insideinfo
```

## Step 4: CoreDNS에 GitLab 호스트 등록

k8s 파드(Argo CD 포함)에서 GitLab 도메인을 찾을 수 있도록 CoreDNS에 등록한다.
이 작업을 안 하면 Argo CD가 GitLab 레포에 연결할 수 없다.

```bash
# <K3S_NODE_IP>=192.168.70.142  <K3S_NODE_HOSTNAME>=hompage-prod
# <GITLAB_IP>=192.168.70.210    <GITLAB_DOMAIN>=gitlab.insideinfo.co.kr
kubectl -n kube-system patch configmap coredns --type merge \
  -p '{"data":{"NodeHosts":"<K3S_NODE_IP> <K3S_NODE_HOSTNAME>\n<GITLAB_IP> <GITLAB_DOMAIN>\n"}}'

kubectl rollout restart deployment coredns -n kube-system
```

확인:

```bash
kubectl rollout status deployment coredns -n kube-system

# DNS 해석 테스트
kubectl run dns-test --rm -it --restart=Never --image=busybox -- nslookup <GITLAB_DOMAIN>  # gitlab.insideinfo.co.kr
```

`<GITLAB_IP>` 가 응답되면 정상.

> **주의**: k3s 재설치 시 CoreDNS ConfigMap이 초기화될 수 있다. 이 경우 이 단계를 다시 실행한다.

## Step 5: k3s 레지스트리 SSL 설정

GitLab 레지스트리가 자체서명 인증서를 사용하므로, k3s containerd가 이미지를 pull할 수 있도록 설정한다.
이 작업을 안 하면 파드가 `ImagePullBackOff` (certificate signed by unknown authority) 에러를 낸다.

```bash
vi /etc/rancher/k3s/registries.yaml
```

아래 내용 입력 후 저장:

```yaml
mirrors:
  <GITLAB_REGISTRY>:                              # gitlab.insideinfo.co.kr:5005
    endpoint:
      - "https://<GITLAB_REGISTRY>"               # https://gitlab.insideinfo.co.kr:5005
configs:
  "<GITLAB_REGISTRY>":                            # gitlab.insideinfo.co.kr:5005
    tls:
      insecure_skip_verify: true
```

k3s 재시작:

```bash
systemctl restart k3s
```

확인:

```bash
systemctl status k3s --no-pager | head -5
cat /etc/rancher/k3s/registries.yaml
```

> **주의**: k3s 재설치 시 이 파일이 삭제될 수 있다. 이 경우 다시 생성 + 재시작한다.

## Step 8: Argo CD Application 배포

서버에서 gitops 레포를 clone한 뒤, 각 앱의 Application 매니페스트를 apply한다.

```bash
# gitops 레포 clone (자체서명 인증서 → sslVerify 비활성화)

cd insideinfo-gitops

# API Application
kubectl apply -f apps/insideinfo-api/argocd/application-dev.yaml

# Web Application
kubectl apply -f apps/insideinfo-web/argocd/application-dev.yaml
```

> Argo CD Web UI에서도 확인 가능: Applications 메뉴에서 `insideinfo-api-dev`, `insideinfo-web-dev` 표시됨.
>
> 파일 위치:
> - `apps/insideinfo-api/argocd/application-dev.yaml`
> - `apps/insideinfo-web/argocd/application-dev.yaml`

## Step 10: 트러블슈팅

### 이미지 pull 실패 (ImagePullBackOff)

```bash
kubectl describe pod -n insideinfo <pod-name>
```

원인: imagePullSecret이 없거나 잘못됨.

조치:
```bash
# secret 확인
kubectl get secret gitlab-registry-secret -n insideinfo

# 재생성
kubectl delete secret gitlab-registry-secret -n insideinfo
kubectl create secret docker-registry gitlab-registry-secret \
  --docker-server=<GITLAB_REGISTRY> \        # gitlab.insideinfo.co.kr:5005
  --docker-username=<GITLAB_USER> \          # qwep0224
  --docker-password=<GITLAB_TOKEN> \         # GitLab Personal Access Token
  -n insideinfo
```

### API 파드 CrashLoopBackOff

DB 연결 실패 가능성:

```bash
kubectl logs -n insideinfo -l app.kubernetes.io/name=insideinfo-api --tail=100
```

확인:
- `<DB_IP>:<DB_PORT>` (192.168.70.142:3306) 에서 k8s 노드 IP의 접근이 허용되어 있는지
- MySQL 서버가 실행 중인지
- Secret의 DB 비밀번호가 정확한지

### HTTPRoute가 동작하지 않음

```bash
# HTTPRoute 상태 확인
kubectl get httproute -n insideinfo -o yaml

# parentRefs 상태에서 "Accepted: True" 확인
# "Accepted: False" 라면 Gateway 설정 문제
```

### git clone SSL 에러

`SSL certificate problem: self signed certificate in certificate chain`

조치:
```bash
git -c http.sslVerify=false clone <REPO_URL>
```
또는 서버 전역 설정:
```bash
git config --global http.sslVerify false
```

### 이미지 pull certificate 에러

`x509: certificate signed by unknown authority`

조치: Step 5 (k3s 레지스트리 SSL 설정) 참고.
`/etc/rancher/k3s/registries.yaml` 생성 후 `systemctl restart k3s`.

### Argo CD에서 Sync 실패

Argo CD Web UI에서 해당 Application 클릭 → 오류 메시지 확인.

흔한 원인:
- gitops 레포 push가 안 됨
- 레포 URL이나 인증 정보 오류
- Helm 차트 문법 오류

## Jenkins 확인 사항

### 버전 관리: 날짜 태그 + latest

Jenkins 파이프라인에서 빌드 시 날짜 기반 태그와 latest를 동시에 push한다.

```groovy
def version = new Date().format("yyyyMMdd-HHmm")   // 예: 20260413-1430

sh "docker build -t ${IMAGE_NAME}:${version} ."
sh "docker tag ${IMAGE_NAME}:${version} ${IMAGE_NAME}:latest"
sh "docker push ${IMAGE_NAME}:${version}"
sh "docker push ${IMAGE_NAME}:latest"
```

| 태그 | 역할 |
|------|------|
| `20260413-1430` | 버전 이력 보관, 롤백 시 사용 |
| `latest` | Argo CD가 항상 최신 이미지 pull |

롤백 시 values 파일의 `image.tag`를 `latest` → `20260413-1430` 등으로 변경 후 push.

### insideinfo-web 빌드 시 build-arg 필수

Jenkins에서 insideinfo-web Docker 이미지를 빌드할 때 `NEXT_PUBLIC_API_URL`을 반드시 전달해야 한다.

```bash
docker build \
  --build-arg NEXT_PUBLIC_API_URL=http://insideinfo-api-dev:8080 \
  -t <GITLAB_REGISTRY>/ax-2/insideinfo-web/insideinfo-web:latest .
  # gitlab.insideinfo.co.kr:5005/ax-2/insideinfo-web/insideinfo-web:latest
```

이유:
- Next.js의 `NEXT_PUBLIC_` 변수는 빌드 시 소스에 인라인됨
- SSR(서버사이드 렌더링)에서 k8s 내부 DNS로 API 호출할 때 사용
- 브라우저는 상대경로 사용 (rewrites 프록시) → 이 값과 무관

### insideinfo-api는 특별한 build-arg 불필요

```bash
docker build -t <GITLAB_REGISTRY>/ax-2/insideinfo-api/insideinfo-api:latest .
# gitlab.insideinfo.co.kr:5005/ax-2/insideinfo-api/insideinfo-api:latest
```

## 외부 IP 연결 이후 할 일

외부 IP가 확보되면 아래 순서로 진행한다.
(상세: `07-외부-IP-나중-반영-가이드.md`)

### 1. k3s 외부 IP 반영

```bash
# /etc/k3s-env 업데이트
sudo sed -i 's/^EXTERNAL_IP=.*/EXTERNAL_IP=<EXTERNAL_IP>/' /etc/k3s-env

# k3s 재반영
sudo bash /opt/k3s-lab/k3s-master.sh
```

### 2. 방화벽 포트 오픈

- `6443/tcp` — kubectl 외부 접속
- `30080/tcp` — 웹 서비스 외부 접속

### 3. 외부 PC hosts 파일 업데이트

```
<EXTERNAL_IP>  insideinfo.local api.insideinfo.local
<EXTERNAL_IP>  grafana.local prometheus.local argocd.local dashboard.local
```

### 4. 외부 접속 확인

```bash
curl http://api.insideinfo.local:30080/actuator/health
curl -I http://insideinfo.local:30080
```

### 5. (선택) 실제 도메인 연결 시

httproute hostname을 실제 도메인으로 변경:

`insideinfo-gitops/apps/insideinfo-api/helm/values/dev/inside-api-values.yaml`:
```yaml
httproute:
  hostnames:
    - api.insideinfo.co.kr  # 실제 도메인
```

`insideinfo-gitops/apps/insideinfo-web/helm/values/dev/inside-web-values.yaml`:
```yaml
httproute:
  hostnames:
    - insideinfo.co.kr  # 실제 도메인
```

gitops push 후 Argo CD가 자동 반영.

### 6. (선택) TLS 인증서

cert-manager + Let's Encrypt 연동은 외부 80/443 포트가 열려야 가능.
Gateway에 HTTPS listener 추가 필요.

### 7. (선택) SealedSecrets로 DB 비밀번호 암호화

현재 `inside-api-values.yaml`에 DB 비밀번호가 평문으로 있음.
운영 환경에서는 kubeseal로 암호화 권장:

```bash
# 현재 Secret 추출
kubectl get secret insideinfo-api-dev-config -n insideinfo -o yaml > plain-secret.yaml

# SealedSecret으로 암호화
kubeseal --format yaml < plain-secret.yaml > sealed-secret.yaml

# gitops 레포에 sealed-secret.yaml 추가
```

## 배포 구조 요약

```
브라우저
  │
  │ http://insideinfo.local:30080
  ▼
Envoy Gateway (NodePort 30080)
  │
  ├─ Host: insideinfo.local     → insideinfo-web-dev:3000
  │                                   │ (rewrites 프록시)
  │                                   ▼
  ├─ Host: api.insideinfo.local → insideinfo-api-dev:8080
  │                                   │
  │                                   ▼
  │                              MySQL (<DB_IP>:<DB_PORT>)
  │
  └─ Argo CD가 gitops 레포 감시 → 변경 시 자동 sync
```
