# 03. 설치 단계별 가이드

## 전체 설치 흐름

```
[1단계] vagrant up                         → VM 3대 생성 (10~20분)
[2단계] k8s-common.sh × 3대                → OS 설정 + containerd + kubeadm (각 5~10분)
[3단계] k8s-master.sh × Master 1대          → 클러스터 초기화 + Calico (3~5분)
[4단계] k8s-worker.sh × Worker 2대          → 클러스터 조인 (각 1분)
[5단계] k8s-master-apps.sh × Master 1대     → 전체 DevOps 스택 설치 (10~15분)
[6단계] 확인                                → kubectl get nodes + Pod 상태
```

총 소요 시간: 약 30~50분 (인터넷 속도에 따라 다름)

> **왜 master.sh와 master-apps.sh를 나눴나요?**
> 워커 노드가 조인된 후에 스택을 설치해야 Pod들이 워커에 분산 스케줄링됩니다.
> 마스터만 있는 상태에서 스택을 설치하면 taint 때문에 Pod가 Pending에 걸리거나,
> taint를 해제해도 모든 부하가 마스터에 집중됩니다.

## 2단계 — 공통 스크립트 실행 (3대 모두)

각 VM에 접속해서 순서대로 실행합니다.

### Master
```bash
vagrant ssh master-node
sudo bash /vagrant/k8s-common.sh
# "공통 설정 완료" 출력 확인
exit
```

### Worker 1
```bash
vagrant ssh worker-node1
sudo bash /vagrant/k8s-common.sh
exit
```

### Worker 2
```bash
vagrant ssh worker-node2
sudo bash /vagrant/k8s-common.sh
exit
```

### k8s-common.sh가 하는 일 (순서대로)

| 단계 | 내용 | 설명 |
|------|------|------|
| [0-1] | root 비밀번호 | `root:1234` |
| [0-2] | 타임존 | Asia/Seoul, NTP 동기화 |
| [0-4] | 기본 패키지 | dnf-utils, iproute-tc, git |
| [0-5] | /etc/hosts | 3대 IP-호스트명 매핑 |
| [0-6] | 방화벽 해제 | firewalld 중지/비활성화 |
| [0-7] | Swap 비활성화 | kubelet 요구사항 |
| [0-8] | SELinux | Permissive 모드 |
| [1-1] | iptables | overlay, br_netfilter 모듈 + sysctl |
| [2] | containerd | Docker repo에서 최신 containerd 설치, cgroup systemd |
| [3] | kubeadm | pkgs.k8s.io v1.34 repo에서 kubelet/kubeadm/kubectl |
| [3-3] | 한글 | glibc-langpack-ko |

> 디스크 확장은 주석 처리되어 있습니다. `generic/rocky9` box는 128GB LVM 동적 디스크이므로 별도 확장이 필요 없습니다.

### 주요 명령어 설명

**`set -e`** — 스크립트 실행 중 어떤 명령이 실패하면 즉시 중단. 디버깅이 쉬워짐.

**iptables 세팅 (overlay, br_netfilter)** — Kubernetes Pod 간 통신에 필수. `br_netfilter`는 bridge 네트워크를 지나는 패킷이 iptables 규칙을 타도록 하고, `ip_forward`는 노드가 패킷을 다른 인터페이스로 전달할 수 있게 함.

**containerd cgroup systemd** — Kubernetes가 기본으로 systemd cgroup을 사용하므로 containerd도 맞춰줘야 함.

## 3단계 — Master 클러스터 초기화

```bash
vagrant ssh master-node
sudo bash /vagrant/k8s-master.sh
```

### k8s-master.sh가 먼저 확인하는 것

`k8s-master.sh`는 시작할 때 `/etc/k8s-env`를 먼저 확인합니다.

- 파일이 있고 필수 값이 정상이라면 그대로 진행
- 파일이 없거나 비정상이라면 같은 디렉터리의 `k8s-common.sh`를 자동 실행
- `k8s-common.sh` 실행 후 `/etc/k8s-env`를 다시 검증하고 나서 설치 진행

즉 `k8s-cleanup.sh`로 정리한 직후에도 아래처럼 바로 다시 시작할 수 있습니다.

```bash
sudo bash /vagrant/k8s-master.sh
```

단, 이 경우에도 `k8s-common.sh` 상단 설정값은 먼저 맞춰놔야 합니다.

### k8s-master.sh가 하는 일

| 단계 | 컴포넌트 | 설치 방식 | 소요 시간 |
|------|---------|----------|----------|
| [4] | kubeadm init + Calico | Manifest | 3~5분 |
| [5] | kubectl 편의기능 | bashrc | 즉시 |

### 완료 시 출력

스크립트가 끝나면 다음 단계 안내가 출력됩니다.
**아직 다른 스크립트를 실행하지 마세요!** 먼저 워커를 조인합니다.

## 4단계 — Worker 조인

### 조인 토큰 확인 (Master에서)
```bash
kubeadm token create --print-join-command
```

출력 예시:
```
kubeadm join 192.168.56.30:6443 --token abcdef.0123456789abcdef \
  --discovery-token-ca-cert-hash sha256:xxxxxxxxxxxxxxxxxxxx
```

이 명령어 전체를 복사합니다.

### Worker 1
```bash
vagrant ssh worker-node1
sudo bash /vagrant/k8s-worker.sh "kubeadm join 192.168.56.30:6443 --token abcdef.0123456789abcdef --discovery-token-ca-cert-hash sha256:xxxx"
exit
```

### Worker 2
```bash
vagrant ssh worker-node2
sudo bash /vagrant/k8s-worker.sh "kubeadm join 192.168.56.30:6443 --token abcdef.0123456789abcdef --discovery-token-ca-cert-hash sha256:xxxx"
exit
```

> 주의: 조인 명령어는 반드시 따옴표(`"..."`)로 감싸야 합니다.

`k8s-worker.sh`도 `/etc/k8s-env`가 없거나 비정상이면 `k8s-common.sh`를 자동 실행한 뒤 조인을 이어갑니다.
즉 워커를 `k8s-cleanup.sh`로 정리한 뒤에는 새 join 명령어만 받아서 바로 다시 붙일 수 있습니다.

### 조인 확인 (Master에서)
```bash
kubectl get nodes
# NAME          STATUS   ROLES           AGE   VERSION
# k8s-master    Ready    control-plane   5m    v1.34.x
# k8s-worker1   Ready    <none>          1m    v1.34.x
# k8s-worker2   Ready    <none>          30s   v1.34.x
```

**3대 모두 Ready 상태를 확인한 후** 다음 단계로 진행합니다.

## 5단계 — 전체 DevOps 스택 설치

```bash
vagrant ssh master-node

```

## 설치 실패 시 트러블슈팅

### kubeadm init 실패
```bash
# 현재 노드 정리 후 재시도
sudo bash /vagrant/k8s-cleanup.sh --yes
sudo bash /vagrant/k8s-master.sh
```

### Helm 설치 실패 (timeout)
```bash
# 특정 Helm 릴리스만 재설치
helm uninstall <릴리스명> -n <네임스페이스>
# k8s-master-apps.sh에서 해당 단계만 복사해서 수동 실행
```

### Pod가 Pending 상태
```bash
kubectl describe pod <pod-name> -n <namespace>
# Events 섹션에서 원인 확인
# 흔한 원인: 리소스 부족, 이미지 Pull 실패, PV 없음
```

### Worker 조인 실패
```bash
# Worker에서 정리 후 재시도
sudo bash /vagrant/k8s-cleanup.sh --yes
# Master에서 새 토큰 생성
kubeadm token create --print-join-command
# 새 토큰으로 재조인
```

## 전체 삭제 / 재설치

### 단일 노드 정리

```bash
sudo bash /vagrant/k8s-cleanup.sh
```

확인 없이 바로 실행하고, `helm`과 `kubeseal`까지 지우려면:

```bash
sudo bash /vagrant/k8s-cleanup.sh --yes --remove-tools
```

옵션 차이:

- `sudo bash /vagrant/k8s-cleanup.sh`
  현재 노드의 kubeadm 기반 Kubernetes와 로컬 설정을 지우고, 실행 전에 확인 질문을 한 번 묻습니다.
- `sudo bash /vagrant/k8s-cleanup.sh --yes`
  위와 같지만 확인 질문 없이 바로 진행합니다.
- `sudo bash /vagrant/k8s-cleanup.sh --remove-tools`
  기본 동작에 더해 `helm`, `kubeseal`도 같이 지웁니다. 확인 질문은 그대로 묻습니다.
- `sudo bash /vagrant/k8s-cleanup.sh --yes --remove-tools`
  확인 질문 없이 바로 진행하고 `helm`, `kubeseal`도 같이 지웁니다.

이 스크립트는 아래를 정리합니다.

- 현재 노드의 kubeadm 클러스터 또는 worker 조인 상태
- `/etc/k8s-env`
- k8s용 sysctl / modules 설정
- `/etc/hosts`의 `k8s-master`, `k8s-worker1`, `k8s-worker2` 항목
- root와 설치 사용자 `.bashrc`에 추가한 kubectl alias
- `/etc/kubernetes`, `/var/lib/kubelet`, `/etc/cni/net.d` 같은 클러스터 잔여 디렉터리

자동으로 되돌리지 않는 항목:

- root 비밀번호
- SELinux 설정
- firewalld 상태
- swap 비활성화
- timezone / locale
- containerd, kubelet, kubeadm, kubectl 패키지 설치 여부

재설치할 때는:

```bash
# Master
sudo bash /vagrant/k8s-master.sh

# Worker
sudo bash /vagrant/k8s-worker.sh "<kubeadm join 명령어>"
```

`k8s-master.sh`, `k8s-worker.sh`는 `/etc/k8s-env`가 없으면 `k8s-common.sh`를 먼저 자동 실행합니다.

### 워커가 있는 클러스터 삭제 순서

1. 마스터에서 워커를 먼저 drain / delete
2. 각 워커에서 `sudo bash /vagrant/k8s-cleanup.sh`
3. 마지막에 마스터에서 `sudo bash /vagrant/k8s-cleanup.sh`

예:

```bash
kubectl drain k8s-worker1 --ignore-daemonsets --delete-emptydir-data
kubectl delete node k8s-worker1

kubectl drain k8s-worker2 --ignore-daemonsets --delete-emptydir-data
kubectl delete node k8s-worker2
```
