## 1. MetalLB — 온프레미스 LoadBalancer

### 역할
클라우드(AWS, GCP 등)에서는 `type: LoadBalancer` 서비스를 만들면 클라우드가 자동으로 External IP를 할당합니다.
온프레미스(VirtualBox, 베어메탈)에서는 이 기능이 없으므로 **MetalLB가 대신 IP를 할당**합니다.

| 항목 | 클라우드 (AWS/GCP) | 온프레미스 (현재) |
|---|---|---|
| LoadBalancer IP 할당 | 클라우드가 자동 | MetalLB가 대신 |
| 방식 | 클라우드 API | L2(ARP) 또는 BGP |
| 비용 | 유료 (LB당 과금) | 무료 |
| 설정 | 클라우드 콘솔 | IPAddressPool CRD |

현재 구성:
```
MetalLB IP Pool: 192.168.56.200 ~ 220 (L2 모드)
    │
    └── Envoy Gateway가 이 풀에서 IP를 받아 외부 트래픽을 수신
```

### 동작 방식 (L2 모드)
MetalLB는 ARP 응답을 통해 "이 IP는 내가 처리한다"고 네트워크에 알립니다.
Windows 호스트가 `192.168.56.200`으로 요청하면, MetalLB가 해당 트래픽을
Gateway Pod가 있는 노드로 전달합니다.

### 상태 확인
```bash
# IP Pool 설정 확인
kubectl get ipaddresspool -n metallb-system

# L2 Advertisement 확인
kubectl get l2advertisement -n metallb-system

# LoadBalancer 서비스에 IP 할당 확인
kubectl get svc -A | grep LoadBalancer

# MetalLB Pod 상태
kubectl get pods -n metallb-system

# 특정 서비스에 할당된 IP 상세 확인
kubectl describe svc -n envoy-gateway-system | grep "LoadBalancer Ingress"
```

### 설정 변경 (IP 대역 변경 시)

**방법 1 — kubectl로 직접 수정 (즉시 반영)**
```bash
kubectl edit ipaddresspool default-pool -n metallb-system
# addresses 부분을 원하는 범위로 수정 후 저장 (:wq)
```

**방법 2 — YAML 파일로 적용**
```yaml
# metallb-pool.yaml
apiVersion: metallb.io/v1beta1
kind: IPAddressPool
metadata:
  name: default-pool
  namespace: metallb-system
spec:
  addresses:
  - 192.168.56.200-192.168.56.250  # 원하는 범위로 변경
---
apiVersion: metallb.io/v1beta1
kind: L2Advertisement
metadata:
  name: default-l2
  namespace: metallb-system
spec:
  ipAddressPools:
  - default-pool
```

```bash
kubectl apply -f metallb-pool.yaml
```

**방법 3 — 클러스터 재설치 시**
`k8s-master-apps.sh`의 [7-1] 섹션에서 IP 범위를 수정 후 스크립트를 실행합니다.

> IP 대역은 반드시 VirtualBox Host-Only Network(192.168.56.0/24) 범위 안에 있어야 하며,
> 노드 IP(30~32)와 겹치지 않아야 합니다.

### 트러블슈팅

**서비스에 External IP가 `<pending>`으로 남아있을 때:**
```bash
# MetalLB controller 로그 확인
kubectl logs -n metallb-system -l app.kubernetes.io/component=controller --tail=20

# IP Pool이 생성되어 있는지 확인
kubectl get ipaddresspool -n metallb-system

# Pool이 없으면 k8s-master-apps.sh의 [7-1] 섹션 참고
```

**IP가 할당됐는데 접속이 안 될 때:**
```bash
# MetalLB speaker가 모든 노드에서 실행 중인지 확인
kubectl get pods -n metallb-system -l app.kubernetes.io/component=speaker -o wide

# Windows에서 해당 IP로 ping 확인
# ping 192.168.56.200
```

### 운영 환경 전환 시
- IP 대역을 실제 네트워크 대역으로 변경
- L2 모드는 같은 서브넷에서만 동작 → 다른 서브넷이면 BGP 모드 사용
- 클라우드 환경에서는 MetalLB 대신 클라우드 LoadBalancer 사용

## 2. Envoy Gateway (Gateway API) — 외부 트래픽 진입점

### 역할
Kubernetes Gateway API의 공식 구현체입니다. 기존 Ingress API를 대체하는 새로운 표준이며,
`Gateway` + `HTTPRoute` 리소스로 트래픽을 라우팅합니다.
NGINX Ingress(kubernetes/ingress-nginx)는 2026년 3월 EOL이므로 Envoy Gateway를 사용합니다.

### Ingress API vs Gateway API 비교

| 항목 | Ingress API (구) | Gateway API (신) |
|---|---|---|
| 리소스 | `Ingress` | `Gateway` + `HTTPRoute` |
| 역할 분리 | 없음 (하나에 다 넣음) | 인프라팀(Gateway) / 앱팀(HTTPRoute) |
| 프로토콜 | HTTP/HTTPS만 | HTTP, gRPC, TCP, UDP |
| 상태 | feature-frozen (더 이상 발전 없음) | 활발히 개발 중 (v1.4) |

### External IP 확인
```bash
# Gateway에 MetalLB가 할당한 IP 확인
kubectl get gateway eg -o jsonpath="{.status.addresses[0].value}"

# 또는 서비스 확인
kubectl get svc -A | grep LoadBalancer
```

### 사용법 — HTTPRoute 생성 (Ingress 대신)

#### 기본 HTTP 라우팅
```yaml
# my-app-route.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-app
spec:
  parentRefs:
  - name: eg       # Gateway 이름 참조
  hostnames:
  - "my-app.local"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: my-app-svc
      port: 80
```
```bash
kubectl apply -f my-app-route.yaml
kubectl get httproute my-app
```

#### 경로 기반 라우팅 (여러 서비스)
하나의 도메인에서 경로로 프론트엔드와 API를 분리합니다.
```yaml
# multi-path-route.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: multi-path
spec:
  parentRefs:
  - name: eg
  hostnames:
  - "example.local"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /api
    backendRefs:
    - name: api-svc
      port: 8080
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: frontend-svc
      port: 80
```
```bash
kubectl apply -f multi-path-route.yaml
kubectl get httproute multi-path
```

#### 트래픽 분할 (Canary/Blue-Green)
새 버전에 10%만 트래픽을 보내고, 문제 없으면 점진적으로 늘리는 방식입니다.
```yaml
# canary-route.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: canary
spec:
  parentRefs:
  - name: eg
  hostnames:
  - "my-app.local"
  rules:
  - backendRefs:
    - name: my-app-v1
      port: 80
      weight: 90    # 기존 버전 90%
    - name: my-app-v2
      port: 80
      weight: 10    # 새 버전 10%
```
```bash
kubectl apply -f canary-route.yaml
kubectl get httproute canary
```

> 적용: `kubectl apply -f <파일>` 후 Windows hosts 파일에 도메인 추가.
> 실전 적용 절차는 04-서비스-접속-가이드.md의 "5. 내 앱에 HTTPRoute 추가하기"를 참고하세요.

### 확인
```bash
# HTTPRoute 목록
kubectl get httproute

# HTTPRoute 상태
kubectl describe httproute my-app

# Gateway 상태
kubectl describe gateway eg
```

### 설정 변경

**Gateway 리스너 수정 (포트, 프로토콜 추가 등)**
```bash
kubectl edit gateway eg -n default
# listeners 섹션에서 포트/프로토콜 추가 가능
```

**HTTPRoute 추가/수정**
```bash
# 새 HTTPRoute 적용
kubectl apply -f my-httproute.yaml

# 기존 HTTPRoute 수정
kubectl edit httproute <이름>

# HTTPRoute 삭제
kubectl delete httproute <이름>
```

**Envoy Gateway Helm 설정 변경**
```bash
# 현재 설정 확인
helm get values eg -n envoy-gateway-system

# 설정 변경 후 업그레이드
helm upgrade eg oci://docker.io/envoyproxy/gateway-helm \
  --version v1.7.1 -n envoy-gateway-system \
  --set <key>=<value>
```

> 초기 설치 설정은 `k8s-master-apps.sh`의 [8] 섹션을 참고하세요.

## 3. cert-manager — TLS 인증서 자동 관리

### 역할
Gateway/Ingress에 HTTPS를 적용할 때 인증서를 자동 발급/갱신합니다.
기본으로 self-signed ClusterIssuer가 생성되어 있습니다.

| 항목 | 수동 인증서 관리 | cert-manager (현재) |
|---|---|---|
| 인증서 발급 | openssl로 직접 생성 | ClusterIssuer가 자동 발급 |
| 갱신 | 만료 전 수동 교체 | 자동 갱신 |
| Let's Encrypt | 직접 ACME 클라이언트 | ClusterIssuer 설정만으로 연동 |
| Secret 관리 | 수동 kubectl create secret | Certificate CRD가 자동 생성 |

### 확인
```bash
# ClusterIssuer 확인
kubectl get clusterissuer

# 발급된 인증서 확인
kubectl get certificate -A
```

### Gateway에 TLS 적용 (Certificate 리소스 사용)
```yaml
# eg-tls-certificate.yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: eg-default-tls
  namespace: default
spec:
  secretName: eg-default-tls
  issuerRef:
    name: selfsigned-issuer
    kind: ClusterIssuer
  dnsNames:
  - "*.local"
  - "my-app.local"
```
```bash
kubectl apply -f eg-tls-certificate.yaml
kubectl get certificate -n default
```
Gateway 리소스의 `tls.certificateRefs`에서 이 Secret을 참조하면 HTTPS가 적용됩니다.

### Let's Encrypt로 전환 (도메인 있을 때)
```yaml
# letsencrypt-issuer.yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: your-email@example.com
    privateKeySecretRef:
      name: letsencrypt-prod
    solvers:
    - http01:
        gatewayHTTPRoute:
          parentRefs:
          - name: eg
            namespace: default
```
```bash
kubectl apply -f letsencrypt-issuer.yaml
kubectl get clusterissuer letsencrypt-prod
```
이후 Certificate 리소스의 `issuerRef.name`을 `letsencrypt-prod`로 변경하면 됩니다.

### 설정 변경

**ClusterIssuer 확인/수정**
```bash
# 현재 ClusterIssuer 목록
kubectl get clusterissuer

# ClusterIssuer 수정 (예: self-signed → Let's Encrypt 전환)
kubectl edit clusterissuer selfsigned-issuer
```

**인증서 재발급**
```bash
# 인증서 삭제하면 cert-manager가 자동 재발급
kubectl delete certificate <이름> -n <네임스페이스>
```

**Helm 설정 변경**
```bash
helm get values cert-manager -n cert-manager
helm upgrade cert-manager jetstack/cert-manager \
  -n cert-manager --set <key>=<value>
```

> 초기 설치 설정은 `k8s-master-apps.sh`의 [9] 섹션을 참고하세요.

## 4. Headlamp — 클러스터 웹 UI

### 역할
Kubernetes Dashboard의 공식 후속 도구입니다 (CNCF Sandbox, SIG UI 관리).
클러스터의 리소스를 웹 브라우저에서 조회/관리할 수 있으며, 플러그인으로 기능을 확장할 수 있습니다.

| 항목 | Kubernetes Dashboard (폐기) | Headlamp (현재) |
|---|---|---|
| 상태 | 2026년 1월 아카이브 | 활발히 개발 중 (SIG UI) |
| 설치 | Helm 차트 404 (repo 삭제됨) | Helm 정상 |
| 아키텍처 | Kong 게이트웨이 의존 (HTTPS만) | 단독 실행 (HTTP:80) |
| 확장성 | 없음 | 플러그인 시스템 (cert-manager, Trivy 등) |
| CRD 지원 | 제한적 | 자동 인식 |

### 접속
브라우저: `http://dashboard.local`
(설치 스크립트에서 HTTPRoute가 자동 설정되어 있음)

### 로그인 토큰 생성
```bash
kubectl -n headlamp create token headlamp
```
출력된 토큰을 브라우저 로그인 화면에 붙여넣기합니다.

### 주요 기능
- Pod, Deployment, Service 등 전체 리소스 조회/편집
- 실시간 로그 확인, 컨테이너 exec 접속
- CRD(Custom Resource) 자동 인식
- RBAC 기반 접근 제어
- 플러그인 시스템 (cert-manager, Trivy 등 확장 가능)

### 설치 주의사항
Helm 차트 0.40.x에서 `-session-ttl` 플래그 호환 문제가 있습니다.
설치 스크립트(`k8s-master-apps.sh`)에서 자동으로 패치하지만,
수동 재설치 시에는 아래 순서로 진행하세요:
```bash
helm upgrade --install headlamp headlamp/headlamp \
  --namespace headlamp --set image.tag="v0.40.1"

kubectl -n headlamp patch deployment headlamp --type='json' -p='[
  {"op": "replace", "path": "/spec/template/spec/containers/0/args", "value": [
    "-in-cluster",
    "-in-cluster-context-name=main",
    "-plugins-dir=/headlamp/plugins"
  ]}
]'
```

### 설정 변경

**Helm 설정 확인/변경**
```bash
# 현재 설정 확인
helm get values headlamp -n headlamp

# 설정 변경 (예: replica 수 변경)
helm upgrade headlamp headlamp/headlamp \
  -n headlamp --set replicaCount=2 --set image.tag="v0.40.1"
# 패치도 다시 필요할 수 있음 (session-ttl 이슈)
```

**Admin 토큰 갱신**
```bash
# 기존 토큰은 만료됨 — 새로 생성
kubectl -n headlamp create token headlamp
```

> 초기 설치 설정은 `k8s-master-apps.sh`의 [10] 섹션을 참고하세요.

## 5. Prometheus + Grafana — 메트릭 모니터링

### 역할
Prometheus가 클러스터의 메트릭(CPU, 메모리, Pod 상태 등)을 수집하고,
Grafana가 이를 시각화합니다. `kube-prometheus-stack`은 이 둘을 한번에 설치합니다.

| 항목 | kubectl top만 사용 | kube-prometheus-stack (현재) |
|---|---|---|
| 메트릭 수집 | 실시간만 (저장 안 됨) | 시계열 DB에 저장 (기본 7일) |
| 시각화 | 터미널 텍스트 | Grafana 대시보드 (그래프/차트) |
| 알림 | 없음 | AlertManager로 Slack/이메일 알림 |
| 커스텀 메트릭 | 불가 | ServiceMonitor로 내 앱 메트릭 수집 |
| Pod/노드 상태 | `kubectl get pods` | 대시보드에서 한눈에 확인 |

### Grafana 접속
브라우저: `http://grafana.local`
ID: `admin` / PW: `admin1234`

### 기본 제공 대시보드
Grafana 왼쪽 메뉴 > Dashboards 에 이미 다음이 포함되어 있습니다:
- Kubernetes / Compute Resources / Cluster
- Kubernetes / Compute Resources / Namespace (Pods)
- Node Exporter / Nodes
- etcd

### 커스텀 ServiceMonitor 추가 (내 앱 메트릭 수집)
```yaml
# my-app-servicemonitor.yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: my-app-monitor
  namespace: monitoring
  labels:
    release: kube-prometheus-stack  # 이 라벨이 있어야 Prometheus가 인식
spec:
  namespaceSelector:
    matchNames: [default]
  selector:
    matchLabels:
      app: my-app
  endpoints:
  - port: metrics
    interval: 30s
```
```bash
kubectl apply -f my-app-servicemonitor.yaml
kubectl get servicemonitor -n monitoring
```

### Prometheus 직접 접속 (PromQL 쿼리)
브라우저: `http://prometheus.local`

### 설정 변경

**Grafana 비밀번호 변경**
```bash
# Helm으로 변경
helm upgrade kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  -n monitoring --set grafana.adminPassword=새비밀번호

# 또는 Grafana UI에서: 좌측 메뉴 > Administration > Users > admin > Change password
```

**Prometheus retention 변경**
```bash
helm upgrade kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  -n monitoring --set prometheus.prometheusSpec.retention=14d
```

**알림 규칙(AlertRule) 추가**
```bash
# 현재 알림 규칙 확인
kubectl get prometheusrule -n monitoring

# 커스텀 알림은 PrometheusRule CRD로 추가
```

**전체 Helm values 확인**
```bash
helm get values kube-prometheus-stack -n monitoring
```

> 초기 설치 설정은 `k8s-master-apps.sh`의 [11] 섹션을 참고하세요.

## 6. Loki + Alloy — 로그 수집/조회

### 역할
Alloy(Promtail 후속)가 각 노드의 컨테이너 로그를 수집하여 Loki에 전송하고,
Grafana에서 로그를 검색/조회할 수 있습니다.

| 항목 | kubectl logs만 사용 | Loki + Alloy (현재) |
|---|---|---|
| 로그 조회 | Pod 하나씩 직접 확인 | Grafana에서 전체 검색 |
| 로그 보존 | Pod 삭제되면 사라짐 | Loki에 저장 (retention 설정 가능) |
| 검색 | 불가 (tail만 가능) | LogQL로 필터/집계/패턴 검색 |
| 여러 Pod | 각각 `kubectl logs` 실행 | `{namespace="default"}` 한 줄로 전체 |
| 수집 에이전트 | 없음 | Alloy DaemonSet (자동 수집) |

| 항목 | Promtail (폐기) | Alloy (현재) |
|---|---|---|
| 상태 | 2026년 3월 EOL | 활발히 개발 중 (공식 후속) |
| 기능 | 로그 수집만 | 로그 + 메트릭 + 트레이스 수집 |
| 설정 언어 | YAML | River (Alloy 자체 DSL) |

### 현재 구성
- **Loki**: SingleBinary 모드 + filesystem 스토리지 (emptyDir)
- **Alloy**: DaemonSet으로 워커 노드마다 실행, Pod 로그 수집
- **인증**: 비활성화 (`auth_enabled: false`) — 멀티테넌시 미사용

```
Worker1: Alloy(수집) ──┐
                       ├──→ Loki(저장) ← 1개 Pod
Worker2: Alloy(수집) ──┘
```

> **학습 환경 주의**: emptyDir 사용으로 Loki Pod 재시작 시 로그가 사라집니다.
> 운영 환경에서는 S3/GCS 등 오브젝트 스토리지로 교체하세요.

### Grafana에 Loki 데이터소스 추가
1. Grafana 접속 > 왼쪽 메뉴 > Connections > Data Sources
2. Add data source > Loki 선택
3. URL: `http://loki-gateway.logging.svc.cluster.local`
4. Save & Test

> `auth_enabled: false`이므로 별도 헤더 설정 없이 바로 연결됩니다.

### 로그 조회
Grafana > Explore > 데이터소스를 Loki로 선택 후:
```
{namespace="default"}                    # default 네임스페이스 전체 로그
{namespace="argocd", app="argocd-server"} # ArgoCD 서버 로그
{pod="my-app-xxxxx"}                      # 특정 Pod 로그
```

### LogQL 예시
```
{namespace="default"} |= "error"              # error 포함된 로그만
{namespace="default"} | json | level="ERROR"   # JSON 로그에서 level 필터
rate({namespace="default"}[5m])                # 5분간 로그 발생률
```

### 수동 재설치 시 참고
Loki Pod가 `/var/loki` 권한 문제로 CrashLoop이 날 수 있습니다.
스크립트에서 emptyDir을 `/var/loki`에 마운트하여 해결하고 있으며,
수동 설치 시에는 `k8s-master-apps.sh`의 [12] 섹션을 참고하세요.

### Alloy 설치 확인
Alloy는 DaemonSet으로 배포되어 각 워커 노드에서 로그를 수집합니다.
```bash
# Alloy Pod 확인 (워커 수만큼 있어야 함)
kubectl get pods -n logging | grep alloy

# Alloy가 없으면 k8s-master-apps.sh의 [12-1] 섹션 실행
# 또는 아래 명령어로 확인:
kubectl get daemonset -n logging
```

### Loki + Alloy 전체 상태 확인
```bash
# Loki Pod (1개 Running)
kubectl get pods -n logging -l app.kubernetes.io/name=loki

# Alloy Pod (워커 노드 수만큼 Running)
kubectl get pods -n logging -l app.kubernetes.io/name=alloy

# Loki Gateway 서비스 (Grafana가 여기로 연결)
kubectl get svc loki-gateway -n logging

# Grafana에서 로그 조회 테스트
# Explore → Loki → {namespace="argocd"} 쿼리
```

### 운영 환경 전환 시 변경 사항

현재는 학습 환경이므로 최소 리소스로 구성되어 있습니다.
운영 환경에서는 아래 항목들을 변경해야 합니다.

#### 학습 → 운영 비교

| 항목 | 학습 (현재) | 운영 |
|------|-----------|------|
| Loki 스토리지 | emptyDir (Pod 재시작 시 유실) | S3 / GCS / MinIO 클러스터 |
| Loki replica | 1개 | 2~3개 (가용성) |
| 인증 (multi-tenancy) | 꺼둠 (`auth_enabled: false`) | 팀별 격리 필요하면 켬 |
| Alloy 설정 | 기본 (전체 Pod 로그 수집) | 라벨 최적화, 불필요한 로그 필터링 |
| 데이터 보존 | 없음 | retention 정책 설정 (예: 30일) |
| 캐시 | 비활성화 | chunks/results 캐시 활성화 |

#### 운영용 Helm values 예시 (S3 스토리지)
```yaml
# loki-prod-values.yaml
deploymentMode: SingleBinary
loki:
  auth_enabled: false  # 또는 true (멀티테넌시 필요 시)
  commonConfig:
    replication_factor: 2
  storage:
    type: s3
    s3:
      endpoint: s3.amazonaws.com
      region: ap-northeast-2
      bucketnames: my-loki-logs
      access_key_id: ${AWS_ACCESS_KEY_ID}
      secret_access_key: ${AWS_SECRET_ACCESS_KEY}
  limits_config:
    retention_period: 720h  # 30일
  schemaConfig:
    configs:
    - from: "2024-04-01"
      store: tsdb
      object_store: s3
      schema: v13
      index:
        prefix: loki_index_
        period: 24h

singleBinary:
  replicas: 2
  persistence:
    enabled: true
    size: 10Gi

minio:
  enabled: false  # 외부 S3 사용

chunksCache:
  enabled: true
resultsCache:
  enabled: true
```

```bash
# 운영 환경 설치
helm upgrade --install loki grafana/loki \
  --namespace logging -f loki-prod-values.yaml
```

> Helm 차트가 같으므로 **values.yaml만 교체**하면 학습→운영 전환이 됩니다.
> LogQL 쿼리, Grafana 데이터소스 설정, Alloy 수집 구조는 동일합니다.

## 7. Sealed Secrets — GitOps Secret 암호화

### 역할
Kubernetes Secret을 암호화하여 Git에 안전하게 저장합니다.
클러스터의 controller만 복호화할 수 있어서 Git에 커밋해도 안전합니다.

| 항목 | 일반 Secret | Sealed Secrets (현재) |
|---|---|---|
| Git 저장 | base64 인코딩뿐 (디코딩 쉬움) → **위험** | 암호화됨 → Git에 커밋 가능 |
| 복호화 | 누구나 `base64 -d`로 가능 | 클러스터의 controller만 가능 |
| GitOps | Secret을 Git에 넣을 수 없음 | SealedSecret을 Git에 안전하게 저장 |
| 워크플로우 | 수동으로 `kubectl create secret` | kubeseal → Git push → ArgoCD 자동 적용 |

### 사용법 — Secret 암호화

#### 1단계: 일반 Secret YAML 생성 (dry-run)
```bash
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password=super-secret-pw \
  --dry-run=client -o yaml > db-secret.yaml
```

#### 2단계: kubeseal로 암호화
```bash
kubeseal --format yaml < db-secret.yaml > db-sealed-secret.yaml
```

#### 3단계: SealedSecret을 Git에 커밋
```bash
# db-sealed-secret.yaml 내용 (안전하게 Git에 저장 가능)
cat db-sealed-secret.yaml
```

#### 4단계: 클러스터에 적용
```bash
kubectl apply -f db-sealed-secret.yaml
# controller가 자동으로 복호화하여 일반 Secret 생성
```

### 확인
```bash
# SealedSecret 리소스 확인
kubectl get sealedsecret

# 복호화된 Secret 확인
kubectl get secret db-credentials -o jsonpath='{.data.password}' | base64 -d
```

### 공개키 백업 (중요!)
```bash
kubeseal --fetch-cert > sealed-secrets-cert.pem
```
이 인증서가 있으면 클러스터 접속 없이도 로컬에서 암호화할 수 있습니다:
```bash
kubeseal --cert sealed-secrets-cert.pem --format yaml < secret.yaml > sealed.yaml
```

### 설정 변경

**Sealed Secrets controller Helm 설정**
```bash
helm get values sealed-secrets -n kube-system
helm upgrade sealed-secrets sealed-secrets/sealed-secrets \
  -n kube-system --set <key>=<value>
```

**키 로테이션 (운영 시 중요)**
```bash
# 현재 키 확인
kubectl get secret -n kube-system -l sealedsecrets.bitnami.com/sealed-secrets-key

# 새 키로 로테이션 시 기존 SealedSecret은 복호화 불가
# 반드시 공개키를 백업한 후 진행
kubeseal --fetch-cert > sealed-secrets-cert-backup.pem
```

> 초기 설치 설정은 `k8s-master-apps.sh`의 [13] 섹션을 참고하세요.

## 8. Argo CD — GitOps 배포

### 역할
Git 저장소를 단일 진실의 원천(Single Source of Truth)으로 삼아,
Git에 push하면 클러스터에 자동으로 배포합니다.

| 항목 | 수동 배포 (kubectl apply) | ArgoCD GitOps (현재) |
|---|---|---|
| 배포 방식 | 사람이 직접 `kubectl apply` | Git push → 자동 sync |
| 상태 관리 | 누가 뭘 바꿨는지 모름 | Git 히스토리 = 배포 히스토리 |
| 롤백 | 이전 YAML 찾아서 수동 적용 | Git revert → 자동 롤백 |
| 환경 차이 | 클러스터와 Git이 다를 수 있음 | selfHeal로 자동 동기화 |
| 멀티 클러스터 | 각각 수동 | ArgoCD 하나로 여러 클러스터 관리 |

### 접속
브라우저: `http://argocd.local`
ID: `admin`
PW:
```bash
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d; echo
```

### UI에서 Git 저장소 연결 (프라이빗 repo만 필요)

> **퍼블릭 repo는 이 단계를 건너뛰세요.** Application 생성 시 URL만 입력하면 바로 접근됩니다.

프라이빗 repo인 경우 인증 정보를 먼저 등록해야 합니다:
1. 좌측 메뉴 **Settings** > **Repositories** > **+ CONNECT REPO**
2. 설정:
   - Connection method: `VIA HTTPS`
   - Repository URL: `https://github.com/your-org/your-repo.git`
   - Username: GitHub 아이디
   - Password: Personal Access Token (GitHub Settings > Developer settings > Tokens에서 생성)
3. **CONNECT** 클릭 → `Successful` 확인

### UI에서 Application 생성 (배포)

1. 좌측 메뉴 **Applications** > **+ NEW APP**
2. **GENERAL** 탭:
   - Application Name: `my-app`
   - Project Name: `default`
   - Sync Policy: dev는 `Automatic`, prod는 `Manual` (수동 승인)
     - ✅ PRUNE RESOURCES (Git에서 삭제하면 클러스터에서도 삭제)
     - ✅ SELF HEAL (클러스터에서 수동 변경하면 Git 상태로 복원)
   - SYNC OPTIONS: ✅ Auto-Create Namespace (네임스페이스 자동 생성)
3. **SOURCE** 탭:
   - Repository URL: `https://github.com/your-org/your-repo.git`
   - Revision: `main` (또는 배포 브랜치)
   - Path: `app-server/helm/back` (**Chart.yaml이 있는 폴더** — templates까지 들어가면 안 됨!)
4. **Helm 설정** (Path 입력 후 Chart.yaml 감지되면 자동으로 나타남):
   - VALUES FILES: `values/dev/api-tester-values.yaml` (**VALUES FILES 칸에 입력**)
   - VALUES: **비워두기** (여기에 파일 경로를 넣으면 에러남)
5. **DESTINATION** 탭:
   - Cluster URL: `https://kubernetes.default.svc` (현재 클러스터)
   - Namespace: `api-tester-dev` (원하는 네임스페이스)
6. **CREATE** 클릭

> **중요: VALUES FILES가 MANIFEST에 반영되었는지 확인**
> CREATE 후 DETAILS > MANIFEST 탭에서 `helm.valueFiles`가 있는지 확인하세요:
> ```yaml
> source:
>   helm:
>     valueFiles:
>       - values/dev/api-tester-values.yaml
> ```
> 없으면 **EDIT**에서 직접 추가해야 합니다. UI에서 입력해도 저장 안 되는 경우가 있습니다.

### Helm 차트 + 환경별 values 구조

```
gitops-repo/
└── app-server/helm/back/
    ├── templates/           ← Helm 템플릿 (공통)
    │   ├── deployment.yaml
    │   ├── service.yaml
    │   ├── configmap.yaml
    │   └── secret.yaml
    ├── Chart.yaml           ← ArgoCD PATH는 여기를 가리켜야 함
    ├── values.yaml          ← 기본값 (모든 키의 기본값 필요)
    └── values/
        └── dev/
            └── api-tester-values.yaml  ← 환경별 오버라이드
```

ArgoCD Application을 환경별로 생성:
```
api-tester-dev  → valueFiles: values/dev/api-tester-values.yaml  (Auto Sync)
api-tester-prod → valueFiles: values/prod/api-tester-values.yaml (Manual Sync)
```

### 배포 확인 (UI)

- Application 카드 클릭 → 리소스 트리가 시각적으로 표시됨
- 각 리소스(Deployment, Service, Pod 등)의 상태를 실시간으로 확인
- **SYNC** 버튼: 수동으로 즉시 동기화
- **REFRESH** 버튼: Git 변경사항 확인
- **ROLLBACK**: History에서 이전 버전으로 롤백

### 배포 흐름

```
1. Git에 values/매니페스트 push
       │
       ▼
2. ArgoCD가 변경 감지 (기본 3분 간격 폴링)
       │
       ▼
3. dev: 자동 Sync / prod: UI에서 SYNC 버튼 클릭
       │
       ▼
4. UI에서 상태 확인 (Healthy / Synced)
```

### 트러블슈팅 — 배포 시 자주 겪는 문제

**Pod가 `InvalidImageName`일 때**
```bash
# 실제 이미지 확인
kubectl get pod -n <네임스페이스> -o jsonpath='{.items[0].spec.containers[0].image}'
```
원인: `values.yaml`(기본값)의 `image.repository`가 빈값(`""`)이고, 환경별 values가 오버라이드 안 됨.
해결: MANIFEST에서 `helm.valueFiles`가 있는지 확인. 없으면 EDIT에서 직접 추가.

**`helm.valueFiles`가 UI에서 저장 안 될 때**
DETAILS > MANIFEST 탭 > EDIT에서 직접 YAML 수정:
```yaml
source:
  repoURL: https://github.com/your-org/your-repo.git
  path: app-server/helm/back
  targetRevision: main
  helm:
    valueFiles:
      - values/dev/api-tester-values.yaml
```

**`repository not accessible: 404 Not Found`일 때**
원인: SOURCE 타입이 `HELM`으로 되어있음 (Helm chart repository로 인식).
해결: SOURCE 타입을 **GIT**으로 변경.

**Pod가 `FailedMount` (Secret not found)일 때**
원인: values의 `secretName`이 실제 생성되는 Secret 이름과 불일치.
Helm `fullname` 기준으로 Secret 이름이 결정됨:
```bash
# 실제 Secret 이름 확인
kubectl get secret -n <네임스페이스>
# values의 secretName을 실제 이름으로 맞추기
```

**VALUES 칸에 파일 경로를 넣으면 에러남**
- `VALUES FILES`: 파일 경로 (예: `values/dev/api-tester-values.yaml`)
- `VALUES`: 인라인 YAML 직접 작성 (보통 비워둠)
- 두 개를 혼동하면 `Values must be a map` 에러 발생

### 설정 변경

**Sync 주기 변경 (기본 3분)**
```bash
kubectl edit configmap argocd-cm -n argocd
# data 섹션에 추가:
#   timeout.reconciliation: 60s  (1분으로 변경)
kubectl rollout restart deployment argocd-repo-server -n argocd
```

**Git 저장소 관리**
UI에서: Settings > Repositories에서 추가/삭제/수정

**insecure 모드 해제 (TLS 활성화)**
```bash
kubectl patch configmap argocd-cmd-params-cm -n argocd --type merge \
  -p '{"data":{"server.insecure":"false"}}'
kubectl rollout restart deployment argocd-server -n argocd
# Gateway의 HTTPS 리스너 + cert-manager 인증서 설정도 필요
```

**ArgoCD ConfigMap 직접 수정**
```bash
# ArgoCD는 Manifest 설치라 Helm values가 없음
kubectl edit configmap argocd-cm -n argocd
kubectl edit configmap argocd-cmd-params-cm -n argocd
kubectl edit configmap argocd-rbac-cm -n argocd
```

> 초기 설치 설정은 `k8s-master-apps.sh`의 [14] 섹션을 참고하세요.

### 참고: YAML로 Application 정의

## 9. Argo CD Image Updater — 이미지 자동 업데이트

### 역할
컨테이너 레지스트리에 새 이미지가 푸시되면 자동으로 Argo CD Application을 업데이트합니다.

| 항목 | 수동 이미지 업데이트 | Image Updater (현재) |
|---|---|---|
| 새 이미지 반영 | YAML에서 tag 직접 수정 → Git push | 레지스트리 감시 → 자동 업데이트 |
| 버전 관리 | 사람이 태그 확인 | semver 규칙으로 자동 선택 |
| CI/CD 연동 | CI에서 Git 커밋까지 해야 함 | CI는 이미지 push만, 나머지 자동 |
| 롤백 | 이전 태그로 수동 변경 | ArgoCD에서 이전 sync로 롤백 |

### 사용법 — ImageUpdater CR 생성 (v1.x CRD 방식)
```yaml
# image-updater-my-app.yaml
apiVersion: argocd-image-updater.argoproj.io/v1alpha1
kind: ImageUpdater
metadata:
  name: my-app-updater
  namespace: argocd
spec:
  applicationRefs:
  - namePattern: "my-app"
  images:
  - alias: "myapp"
    imageName: "your-registry/my-app:1.0"
    commonUpdateSettings:
      updateStrategy: semver
      versionConstraint: "~1.0"
```
```bash
kubectl apply -f image-updater-my-app.yaml
kubectl get imageupdater -n argocd
```

### 확인
```bash
# Image Updater 로그 확인
kubectl -n argocd logs -l app.kubernetes.io/name=argocd-image-updater -f

# 업데이트된 이미지 확인
kubectl get application my-app -n argocd -o yaml | grep image
```

### 프라이빗 레지스트리 인증
```bash
kubectl -n argocd create secret docker-registry my-registry-creds \
  --docker-server=your-registry.com \
  --docker-username=user \
  --docker-password=pass
```
ImageUpdater CR에서 참조:
```yaml
images:
- alias: "myapp"
  imageName: "your-registry.com/my-app:1.0"
  commonUpdateSettings:
    pullSecret: secret:argocd/my-registry-creds
```

### 설정 변경

**Image Updater 로그 레벨 변경**
```bash
kubectl edit configmap argocd-image-updater-config -n argocd
# log.level: debug  (기본: info)
kubectl rollout restart deployment argocd-image-updater -n argocd
```

**업데이트 주기 변경**
```bash
kubectl edit configmap argocd-image-updater-config -n argocd
# registries.conf 섹션에서 interval 설정
```

> 초기 설치 설정은 `k8s-master-apps.sh`의 [15] 섹션을 참고하세요.

## 전체 흐름 예시: 앱 배포부터 모니터링까지

### 1. 앱 코드 푸시 → Docker 이미지 빌드
```bash
docker build -t your-registry/my-app:1.0.0 .
docker push your-registry/my-app:1.0.0
```

### 2. K8s 매니페스트 작성 (Git 저장소)
```
gitops-repo/
├── manifests/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── ingress.yaml
└── sealed-secrets/
    └── db-credentials.yaml   # kubeseal로 암호화된 Secret
```

### 3. Argo CD가 자동 배포
Git에 push하면 Argo CD가 감지하고 클러스터에 자동 sync합니다.

### 4. Image Updater가 새 버전 감지
레지스트리에 `1.0.1`이 올라오면 자동으로 이미지 업데이트합니다.

### 5. HTTPRoute로 외부 접속
MetalLB IP → Envoy Gateway → HTTPRoute → Service → Pod 순서로 트래픽이 흐릅니다.
Windows hosts 파일에 도메인을 등록하면 `http://my-app.local`로 접속합니다.

### 6. Grafana에서 모니터링
- 메트릭: Prometheus 데이터소스 (자동 연결)
- 로그: Loki 데이터소스 (수동 추가 필요)

## 서비스 접속 요약 (HTTPRoute 도메인 라우팅)

모든 서비스는 Envoy Gateway의 HTTPRoute를 통해 도메인으로 접속합니다.
port-forward는 필요 없으며, VM 재시작 후에도 설정이 유지됩니다.

| 서비스 | URL | 인증 |
|--------|-----|------|
| Grafana | http://grafana.local | admin / admin1234 |
| ArgoCD | http://argocd.local | admin / (secret에서 확인) |
| Headlamp | http://dashboard.local | 토큰 (create token) |
| Prometheus | http://prometheus.local | 없음 |

> Windows hosts 파일에 Gateway IP → 도메인 매핑이 등록되어 있어야 합니다.
> 설정 방법은 04-서비스-접속-가이드.md를 참고하세요.
