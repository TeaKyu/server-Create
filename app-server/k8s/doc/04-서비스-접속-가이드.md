# 04. 서비스 접속 가이드

## 접속 방식: HTTPRoute 도메인 라우팅

모든 서비스는 Envoy Gateway의 HTTPRoute를 통해 도메인으로 접속합니다.
port-forward는 필요 없으며, VM 재시작 후에도 유지됩니다.

```
브라우저 → hosts 파일 (도메인 → IP) → MetalLB IP → Envoy Gateway → HTTPRoute → Service → Pod
```

---

## 1. 사전 설정: Windows hosts 파일

### Gateway IP 확인 (Master에서)
```bash
kubectl get gateway eg -o jsonpath='{.status.addresses[0].value}'
# 예: 192.168.56.200
```

### hosts 파일 수정
1. 메모장을 **관리자 권한**으로 실행
2. 파일 열기: `C:\Windows\System32\drivers\etc\hosts`
3. 아래 내용 추가 (IP는 위에서 확인한 값):

```
192.168.56.200  grafana.local
192.168.56.200  prometheus.local
192.168.56.200  argocd.local
192.168.56.200  dashboard.local
```

4. 저장 후 브라우저에서 바로 접속 가능

> Mac에서 원격으로 접속하는 경우 `PowerShell.txt`의 포트포워딩 설정을 참고하세요.

## 2. 서비스별 접속 정보

| 서비스 | URL | 인증 |
|--------|-----|------|
| Grafana | http://grafana.local | admin / admin1234 |
| Prometheus | http://prometheus.local | 없음 |
| Argo CD | http://argocd.local | admin / (아래 명령어로 확인) |
| Headlamp | http://dashboard.local | Bearer Token (아래 명령어로 생성) |

### ArgoCD 초기 비밀번호
```bash
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d; echo
```

### Headlamp 로그인 토큰
```bash
kubectl -n headlamp create token headlamp
```

## 3. Grafana 설정

### 기본 대시보드
접속 후 좌측 메뉴 > Dashboards에 이미 Kubernetes 대시보드가 포함되어 있습니다:
- Kubernetes / Compute Resources / Cluster
- Kubernetes / Compute Resources / Namespace (Pods)
- Node Exporter / Nodes

### Loki 데이터소스 추가 (로그 조회)
1. 좌측 메뉴 > Connections > Data Sources
2. Add data source > Loki 선택
3. URL: `http://loki-gateway.logging.svc.cluster.local`
4. Save & Test → "Data source successfully connected" 확인

> **로그가 안 보이면?** Alloy(로그 수집 에이전트)가 실행 중인지 확인하세요.
> `kubectl get pods -n logging -l app.kubernetes.io/name=alloy`
> 없으면 05-컴포넌트별-사용법-가이드.md의 "Loki + Alloy" 섹션을 참고하세요.

### 로그 조회
좌측 메뉴 > Explore > 상단 데이터소스를 "Loki"로 변경
```
{namespace="default"}                     # 전체 로그
{namespace="argocd", app="argocd-server"} # ArgoCD 로그
{pod="my-app-xxxxx"} |= "error"           # 특정 Pod에서 error 검색
```

## 4. 접속 안 될 때 빠른 확인

```bash
# 1. 노드 상태
kubectl get nodes

# 2. Gateway IP 할당 확인
kubectl get gateway eg -o jsonpath='{.status.addresses[0].value}'

# 3. HTTPRoute 목록
kubectl get httproute

# 4. 문제 서비스 Pod 상태
kubectl get pods -n <네임스페이스>
```

> 각 컴포넌트별 상세 트러블슈팅은 05-컴포넌트별-사용법-가이드.md를 참고하세요.

## 5. 내 앱에 HTTPRoute 추가하기

앱을 배포한 후 외부에서 도메인으로 접속하려면 3단계가 필요합니다.

### 5-1. 앱의 Service 확인
먼저 배포된 앱의 서비스 이름과 포트를 확인합니다.
```bash
kubectl get svc -n default
# NAME                  TYPE        CLUSTER-IP      PORT(S)
# insideinfo-web-svc    ClusterIP   10.96.xxx.xxx   3000/TCP
```

### 5-2. HTTPRoute 생성
```yaml
# my-app-route.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-app
  namespace: default
spec:
  parentRefs:
  - name: eg              # Gateway 이름 (고정)
  hostnames:
  - "my-app.local"        # 브라우저에서 접속할 도메인
  rules:
  - backendRefs:
    - name: my-app-svc    # 위에서 확인한 Service 이름
      port: 3000           # Service 포트
```

```bash
kubectl apply -f my-app-route.yaml

# 확인
kubectl get httproute my-app
```

> **다른 네임스페이스에 앱이 있는 경우** backendRef에 `namespace`를 추가하고,
> 해당 네임스페이스에 ReferenceGrant가 필요합니다.
> `k8s-master-apps.sh`의 [16-1] 섹션을 참고하세요.

### 5-3. Windows hosts 파일에 도메인 추가

Gateway IP를 확인하고:
```bash
kubectl get gateway eg -o jsonpath='{.status.addresses[0].value}'
# 예: 192.168.56.200
```

`C:\Windows\System32\drivers\etc\hosts`에 추가:
```
192.168.56.200  my-app.local
```

Mac에서 접속하는 경우 Mac의 `/etc/hosts`에도 Windows IP로 추가:
```bash
sudo bash -c 'echo "<WINDOWS_IP>  my-app.local" >> /etc/hosts'
```

### 접속 확인
브라우저에서 `http://my-app.local` 로 접속.

접속이 안 되면:
```bash
# HTTPRoute 상태 확인 (Accepted: True 여야 함)
kubectl describe httproute my-app

# 앱 Pod가 Running인지 확인
kubectl get pods -n default

# Service → Pod 연결 확인
kubectl get endpoints my-app-svc
```

### 경로 기반 라우팅 예시 (프론트엔드 + API)
하나의 도메인에서 경로로 서비스를 분리할 수도 있습니다.
```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-app-routes
spec:
  parentRefs:
  - name: eg
  hostnames:
  - "my-app.local"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /api
    backendRefs:
    - name: api-svc
      port: 8080
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: frontend-svc
      port: 3000
```

### 트래픽 분할 예시 (Canary / Blue-Green 배포)
새 버전을 배포할 때 트래픽을 비율로 나눠서 점진적으로 전환할 수 있습니다.
```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-app-canary
spec:
  parentRefs:
  - name: eg
  hostnames:
  - "my-app.local"
  rules:
  - backendRefs:
    - name: my-app-v1    # 기존 버전 (90%)
      port: 80
      weight: 90
    - name: my-app-v2    # 새 버전 (10%)
      port: 80
      weight: 10
```
```bash
# 적용
kubectl apply -f my-app-canary.yaml

# 새 버전 검증 후 100%로 전환
kubectl edit httproute my-app-canary
# weight: v1=0, v2=100 으로 변경
```

> Envoy Gateway 컴포넌트 설정 변경은 05-컴포넌트별-사용법-가이드.md의 "2. Envoy Gateway" 섹션을 참고하세요.
